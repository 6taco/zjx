# 电子证书跨机构可信验证平台智能合约设计说明

## 1. 文档概述

本文档用于说明电子证书跨机构可信验证平台中智能合约的设计目标、数据结构、核心方法和实现思路。智能合约是本系统实现“可信登记”和“防篡改验证”的关键组成部分，其主要职责不是承载全部业务逻辑，而是保存最关键、最需要可信保证的证书索引信息。

本项目智能合约采用 Solidity 编写，通过 Hardhat 进行编译和部署，并由前端通过 MetaMask 发起交易写入区块链，后端通过 RPC 完成链上只读查询。

## 2. 智能合约设计目标

智能合约在本系统中的设计目标主要包括：

- 为每一份正式发布证书提供链上可信登记入口
- 保存证书的关键索引信息
- 提供链上只读查询能力
- 保证同一证书内容不能重复登记
- 提供链上访问控制，仅授权发行人可写入记录
- 支持证书撤销标记，增强证书生命周期管理
- 与 MySQL 和 IPFS 共同构成“链上索引 + 链下业务 + 文件分布式存储”的混合架构

从系统设计角度看，智能合约不负责用户注册、草稿审批、OCR 识别、授权码管理等复杂业务，但引入了基础的链上访问控制机制，将主要业务治理继续留在后端。这样能够在保证链上写入权限可控的同时，有效降低合约复杂度和链上存储成本。

## 3. 设计原则

### 3.1 链上最小化存储原则

由于区块链存储成本较高，且链上数据一经写入难以修改，因此本项目采用链上最小化存储策略，只将最关键的可信索引写入链上，而不直接把完整证书文件或复杂业务字段存入区块链。

链上保存内容包括：

- 证书哈希 `certHash`
- IPFS 哈希 `ipfsHash`
- 发布者地址 `issuer`
- 上链时间 `timestamp`
- 撤销标记 `revoked`

这样设计能够满足电子证书可信验证的核心需求，同时避免高额链上开销。

### 3.2 链下业务治理原则

系统中的复杂业务逻辑，如用户注册登录、角色权限判断、普通用户草稿提交、管理员审批发布、验证日志记录和授权码管理，全部由后端完成。智能合约只负责可信登记与查询，不直接承担业务治理职责。

这种设计有以下优势：

- 减少智能合约逻辑复杂度
- 降低链上执行成本
- 提升整体开发效率
- 便于后续维护和调试

### 3.3 前端发起交易原则

本系统选择由前端通过 MetaMask 发起合约写入，而不是由后端统一代签。这样设计的主要原因包括：

- 用户显式确认链上操作
- 后端无需保存用户私钥
- 更符合常见区块链应用使用方式
- 增强演示和答辩过程中的可视化效果

## 4. 合约总体结构设计

本项目智能合约名称为 `CertificateRegistry`，主要由以下部分组成：

- 证书结构体定义
- 状态变量（合约拥有者、授权发行人映射、证书计数、证书映射）
- 访问控制修饰符（`onlyOwner`、`onlyAuthorized`）
- 证书登记方法（单条与批量）
- 证书撤销方法
- 证书查询方法（基础查询、扩展查询、存在性检测）
- 权限管理方法（授权、撤权、转让所有权）
- 事件定义

合约整体设计强调“结构清晰、权限可控、可验证”，适合作为毕业设计中的可信登记核心模块。

## 5. 数据结构设计

### 5.1 Certificate 结构体设计

为了保存链上证书索引信息，合约定义了 `Certificate` 结构体，其字段包括：

- `certHash`：证书内容哈希
- `ipfsHash`：证书文件在 IPFS 中的哈希地址
- `issuer`：发起写入的区块链地址
- `timestamp`：证书写入区块链的时间戳
- `revoked`：证书撤销标记

该结构体的设计意义如下：

- `certHash` 用于唯一标识一份证书内容
- `ipfsHash` 用于关联原始证书文件
- `issuer` 用于记录链上发布者身份
- `timestamp` 用于提供上链时间依据
- `revoked` 用于标记证书是否已被撤销

通过这五项数据，系统能够支撑“文件真实存在、索引可信登记、可查询发布时间和发布地址、可检测撤销状态”的核心需求。

### 5.2 状态变量与映射存储设计

合约定义了以下状态变量：

- `owner`：合约拥有者地址，部署时自动设置为部署者
- `authorizedIssuers`：授权发行人白名单映射（`mapping(address => bool)`），部署时自动将拥有者加入
- `certificateCount`：链上已登记证书总数
- `certificates`：证书映射存储（`mapping(string => Certificate)`），以 `certHash` 作为键

证书映射设计有以下优点：

- 能以证书哈希快速检索链上记录
- 查询复杂度低，适合只读验证场景
- 能保证一份证书哈希对应唯一记录
- 适合电子证书这种按摘要唯一索引的业务模型

授权发行人白名单机制使得仅合约拥有者或被授权地址可以写入链上记录，从而在链上层面提供了基础权限保护。

## 6. 核心方法设计

### 6.1 证书登记方法设计

合约提供 `storeCertificate` 方法，用于将证书哈希和 IPFS 哈希写入区块链。该方法由外部账户调用，调用时需传入以下参数：

- `certHash`
- `ipfsHash`

该方法受 `onlyAuthorized` 修饰符保护，仅合约拥有者或授权发行人可以调用。其执行流程如下：

1. 校验调用者是否为授权地址
2. 校验证书哈希是否为空
3. 校验 IPFS 哈希是否为空
4. 根据 `certHash` 查询当前映射中是否已存在记录
5. 若已存在记录，则拒绝写入
6. 若不存在记录，则构造新的 `Certificate` 结构体，`revoked` 初始为 `false`
7. 将发布者地址设置为 `msg.sender`
8. 将当前区块时间写入 `timestamp`
9. 更新 `certificateCount` 计数
10. 保存记录并触发 `CertificateStored` 事件

该方法的核心作用在于：将一份正式证书的可信索引写入区块链，并利用区块链不可篡改特性增强可信性。

### 6.2 防重复写入设计

为了避免同一份证书重复登记，合约在写入前会检查当前 `certHash` 是否已存在记录。若对应记录中 `certHash` 非空，则视为已登记，直接拒绝再次写入。

这一机制的意义在于：

- 防止同一证书被反复上链
- 保证链上证书索引的一致性
- 简化后续验证逻辑
- 避免重复数据污染链上状态

由于系统在后端发布阶段也会对数据库中的 `cert_hash` 做重复校验，因此形成了“链下校验 + 链上校验”的双重保护机制。

### 6.3 证书查询方法设计

合约提供 `verifyCertificate` 方法，用于根据证书哈希读取链上登记信息。该方法接收一个参数：

- `certHash`

其执行逻辑如下：

1. 根据 `certHash` 从映射中读取对应结构体
2. 检查该记录是否存在
3. 若不存在，则返回错误
4. 若存在，则返回 `ipfsHash`、`issuer` 和 `timestamp`

之所以不再返回 `certHash` 本身，是因为查询输入已经明确给出了证书哈希，链下系统也已经掌握该值，因此只需返回额外的链上信息即可。

该查询方法为系统后端提供了只读验证能力，是证书链上验证功能的关键基础。

### 6.4 批量登记方法设计

合约提供 `storeCertificateBatch` 方法，支持在一笔交易中登记多份证书（上限 50 条）。该方法同样受 `onlyAuthorized` 保护，执行时会校验数组长度一致性，并自动跳过已存在的 `certHash`，避免整个交易回滚。

### 6.5 证书撤销方法设计

合约提供 `revokeCertificate` 方法，用于将已登记证书标记为撤销状态。仅证书的原始发布者或合约拥有者可以执行撤销。撤销操作将 `revoked` 标记设置为 `true`，并触发 `CertificateRevoked` 事件。已撤销的证书不能重复撤销。

### 6.6 扩展查询方法设计

合约提供 `verifyCertificateEx` 方法，在基础查询返回结果基础上额外返回 `revoked` 状态，便于调用方判断证书是否已被撤销。

合约还提供 `certificateExists` 方法，用于快速检查某个 `certHash` 是否已存在链上记录，返回布尔值。

### 6.7 权限管理方法设计

合约提供以下权限管理方法，均受 `onlyOwner` 修饰符保护：

- `authorizeIssuer`：将指定地址加入授权发行人白名单
- `revokeIssuer`：将指定地址从授权发行人白名单中移除（不能移除拥有者）
- `transferOwnership`：将合约所有权转让给新地址，新拥有者自动加入授权白名单

这些方法使得合约能够在链上层面对写入权限进行管理，而不依赖外部系统。

## 7. 事件设计

### 7.1 证书登记事件

为了便于链上行为追踪和调试，合约在成功登记证书时触发 `CertificateStored` 事件，事件中包含以下信息：

- `certHash`
- `ipfsHash`
- `issuer`
- `timestamp`

该事件设计具有以下作用：

- 便于监听链上登记行为
- 有助于调试和问题排查
- 便于后续扩展区块链浏览器展示或日志分析能力

虽然当前系统主要通过交易哈希回写数据库来跟踪上链结果，但事件机制仍是合约设计中的重要组成部分。

### 7.2 其他事件

合约还定义了以下事件：

- `CertificateRevoked`：证书被撤销时触发，包含证书哈希、撤销者地址和时间戳
- `IssuerAuthorized`：新发行人被授权时触发
- `IssuerRevoked`：发行人授权被撤销时触发
- `OwnershipTransferred`：合约所有权转让时触发

这些事件为链上行为追踪和审计提供了完整的日志支撑。

## 8. 合约实现逻辑分析

### 8.1 为什么只存证书哈希和 IPFS 哈希

完整证书文件通常体积较大，直接存储在链上会带来以下问题：

- 存储成本高
- 查询效率低
- 不利于系统扩展
- 会增加不必要的链上负担

因此，本系统选择只在链上保存证书哈希和 IPFS 哈希，将原始文件放到 IPFS 中。这种方式具有以下优点：

- 链上记录更轻量
- 原始文件仍可通过 IPFS 获取
- 证书真伪仍可通过文件哈希比对判断
- 系统整体架构更合理

### 8.2 关于合约中的权限设计

当前合约已引入基础的链上访问控制，通过 `owner` 和 `authorizedIssuers` 白名单机制约束写入权限。但更复杂的业务权限（如用户角色判断、草稿审批流程、授权码管理等）仍由后端处理，以避免过度增加合约复杂度和交易成本。

这种“链上基础权限 + 链下业务治理”的组合方式既能体现区块链权限控制价值，又能保证系统工程可行性。

### 8.3 为什么采用只读验证接口

在验证场景中，用户需要频繁核验证书是否存在链上记录。若每次验证都发起链上交易，不仅成本高，而且没有必要。因此系统采用合约只读方法进行验证查询，这种方式具有以下优势：

- 无需支付交易费用
- 查询速度更快
- 更适合高频验证场景
- 更符合证书核验业务的使用需求

## 9. 合约与系统其他模块的协同设计

### 9.1 与前端的协同

前端在管理员直接发布或草稿审批完成后，会获取后端返回的 `certHash` 和 `ipfsHash`，然后通过 ethers 调用合约中的 `storeCertificate` 或 `storeCertificateBatch` 方法完成链上登记。交易确认后，前端再把交易哈希回写给后端。

这种协同方式实现了：

- 链下业务发布先完成
- 链上登记后补充可信性
- 钱包交互过程由用户确认

### 9.2 与后端的协同

后端在链上验证时，不直接写链，而是通过 RPC 调用 `verifyCertificate` 只读方法，获取链上登记信息，并结合数据库中的业务信息返回给前端。

这种协同方式使得：

- 后端能够整合链上与链下数据
- 验证结果更完整
- 前端不需要直接处理复杂链上查询逻辑

### 9.3 与 IPFS 的协同

原始证书文件和 OCR 元数据由后端上传到 IPFS，返回 `ipfsHash` 后再参与链上写入。因此链上合约实际上保存的是对 IPFS 文件的可信引用，而不是文件本身。

这种设计构成了完整的数据协同链路：

原始文件 → IPFS 存储 → 返回 ipfsHash → 合约写入 certHash 与 ipfsHash → 后续按哈希验证

## 10. 合约设计优点

本项目智能合约设计具有以下优点：

- 结构清晰，职责分明，便于理解和实现
- 仅保存关键索引，链上成本较低
- 提供重复写入保护
- 支持单条和批量登记
- 支持证书撤销标记
- 通过白名单机制控制链上写入权限
- 支持快速只读验证和存在性检测
- 易于与前后端和 IPFS 协同
- 适合作为毕业设计中的可信存证核心模块

## 11. 合约设计局限性

虽然当前合约已实现登记、查询、撤销和基础访问控制，能够较好满足项目原型需求，但仍存在一定局限性：

- 未支持证书版本管理或字段更新
- 链上权限仍为基础白名单模式，未引入更细粒度的机构治理
- 未对更多链上审计字段进行扩展
- 当前更适合原型系统和教学演示，而非大规模生产应用

这些局限性并不影响毕业设计目标，但可以作为论文中的后续优化方向进行说明。

## 12. 后续优化方向

若未来进一步完善智能合约，可考虑以下方向：

- 支持证书版本管理和字段更新
- 引入更细粒度的链上机构治理模型
- 优化字符串存储方式以进一步降低 gas 消耗
- 扩展链上审计字段

## 13. 本章小结

本文档对电子证书跨机构可信验证平台中的智能合约设计进行了说明。系统采用 `CertificateRegistry` 合约，以证书哈希为唯一索引，记录证书哈希、IPFS 哈希、发布者地址、上链时间和撤销状态，并提供单条与批量登记、撤销、多种查询和权限管理能力。该设计遵循链上最小化存储原则，将复杂业务逻辑保留在链下处理，同时通过链上白名单机制控制写入权限，从而在可信性、权限可控性和运行成本之间取得平衡。

整体来看，本项目智能合约结构清晰、职责分明，能够有效支撑电子证书可信发布与验证场景，是系统中实现区块链可信索引能力的核心组成部分。
