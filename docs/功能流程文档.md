# 电子证书跨机构可信验证平台功能流程文档

## 1. 文档目的

本文档用于说明电子证书跨机构可信验证平台的主要功能模块、参与角色以及核心业务流程。系统围绕证书发布、链上登记、真实性验证和跨机构授权验证展开，通过前端页面、后端服务、MySQL、IPFS 和区块链智能合约共同完成电子证书的可信管理。

## 2. 系统功能概述

本系统主要实现以下功能：

1. 用户注册与登录
2. 邮箱验证码校验
3. 证书 OCR 智能识别
4. 证书草稿提交
5. 管理员审批发布
6. 证书正式发布与 IPFS 存储
7. MetaMask 发起链上登记
8. 证书链上验证
9. 证书文件真实性验证
10. 跨机构授权码生成与授权验证
11. 验证日志记录与审计

系统整体业务闭环为：

用户注册登录 → 证书录入 → 草稿提交或直接发布 → 文件上传 IPFS → 证书摘要写入数据库 → MetaMask 发起上链 → 链上验证或文件验真 → 授权控制 → 日志留痕

## 3. 角色与权限

### 3.1 普通用户

普通用户的主要功能包括：

- 注册并登录系统
- 上传证书图片并进行 OCR 识别
- 填写证书信息
- 上传原始证书文件
- 提交证书草稿
- 使用验证模块查询证书状态或进行文件验真

普通用户不能直接发布正式证书，只能进入草稿审批流程。

### 3.2 机构管理员

机构管理员的主要功能包括：

- 登录系统
- 直接发布证书
- 查看本机构草稿箱
- 审批普通用户提交的草稿
- 查看和管理本机构证书

机构管理员既可以直接发布证书，也可以审批草稿。

### 3.3 总管理员

总管理员在管理员权限基础上，额外拥有以下能力：

- 全局范围管理证书
- 生成跨机构验证授权码
- 控制跨机构验证结果的可见范围

## 4. 用户注册与登录流程

### 4.1 用户注册流程

用户注册流程如下：

1. 用户输入邮箱地址并请求发送验证码
2. 后端校验邮箱格式和发送频率
3. 系统生成验证码并发送到用户邮箱
4. 用户填写用户名、密码、邮箱、验证码、机构和角色信息
5. 后端校验验证码是否正确且未过期
6. 系统对密码进行加密处理
7. 用户信息写入数据库
8. 注册成功并返回结果

该流程的核心目的是确保用户身份有效，并保证账户数据安全存储。

### 4.2 用户登录流程

用户登录流程如下：

1. 用户输入用户名和密码
2. 后端查询用户信息
3. 系统校验密码是否正确
4. 校验通过后生成 JWT 登录令牌
5. 前端保存令牌并进入系统工作台
6. 后续请求统一携带令牌完成身份认证

该流程的核心目的是建立合法登录态，为后续业务操作提供身份凭证。

## 5. 证书发布流程

证书发布是系统核心业务流程，根据角色不同分为普通用户草稿发布流程和管理员直接发布流程。

### 5.1 OCR 辅助录入流程

为了减少人工输入工作量，系统支持 OCR 识别。

具体流程如下：

1. 用户上传证书图片
2. 前端向后端发起 OCR 请求
3. 后端调用 OCR 服务识别图片文字
4. 系统提取证书名称、持证人、证书编号、发证机构和日期等字段
5. 前端将识别结果回填到表单
6. 用户检查并补充修正识别内容

该流程的作用是提高证书录入效率，降低手工输入错误率。

### 5.2 普通用户草稿发布流程

普通用户发布证书时采用先提交、后审批的方式。

具体流程如下：

1. 普通用户登录系统
2. 进入证书发布页面
3. 填写证书名称、持证人、证书编号、签发机构、证书类别、颁发日期等信息
4. 上传原始证书文件
5. 提交发布请求
6. 后端识别当前角色为普通用户
7. 系统校验用户是否填写所属机构
8. 后端将证书信息和文件数据写入草稿表
9. 草稿状态设置为待审批
10. 系统返回草稿提交成功结果

该流程结束后，证书尚未正式发布，也尚未上链，需要等待管理员审批。

### 5.3 管理员直接发布流程

管理员和总管理员可以直接发布正式证书。

具体流程如下：

1. 管理员登录系统
2. 填写证书信息并上传原始证书文件
3. 后端接收文件并计算文件 SHA256 哈希
4. 系统检查数据库中是否已存在相同哈希的证书
5. 若存在重复内容，则终止发布
6. 若不存在重复内容，则将证书文件上传到 IPFS
7. 若存在 OCR 文本，则同时上传 OCR 元数据到 IPFS
8. 系统将证书业务信息写入正式证书表
9. 后端返回证书编号、证书哈希和 IPFS 哈希
10. 前端连接 MetaMask 钱包
11. 用户在钱包中确认交易
12. 前端调用智能合约完成证书上链登记
13. 区块链交易确认后，系统回写交易哈希
14. 发布流程结束

该流程实现了业务录入、文件存储、数据入库和链上登记的完整闭环。

## 6. 草稿审批流程

草稿审批流程用于处理普通用户提交的待发布证书。

具体流程如下：

1. 机构管理员登录系统
2. 进入草稿箱页面
3. 系统查询当前管理员所属机构下的待审批草稿
4. 前端展示草稿列表
5. 管理员选择某条草稿并点击确认发布
6. 后端先将草稿状态改为处理中
7. 系统读取草稿中的证书信息和原始文件
8. 后端复用正式发布逻辑生成正式证书
9. 证书文件上传 IPFS，证书信息写入正式证书表
10. 草稿状态更新为已审批
11. 系统记录正式证书 ID 和证书哈希
12. 前端继续发起链上登记
13. 上链成功后保存交易哈希
14. 流程结束

该流程体现了普通用户可提交、管理员掌控最终发布权的业务逻辑。

## 7. 证书查询与管理流程

系统支持用户按权限范围查看和管理证书。

具体流程如下：

1. 用户进入证书列表页面
2. 输入证书编号、持证人、证书类别或证书哈希等查询条件
3. 前端提交查询请求
4. 后端根据当前用户角色和机构范围过滤数据
5. 数据库返回符合条件的证书列表
6. 前端分页展示结果
7. 管理员可进一步查看详情、修改证书信息或删除证书

该流程实现了证书数据的检索、维护与权限控制。

## 8. 证书链上验证流程

系统支持按证书编号、数据库 ID 或证书哈希进行链上验证。

具体流程如下：

1. 用户进入证书验证页面
2. 输入证书编号或证书哈希
3. 前端向后端发送验证请求
4. 后端解析输入值并定位目标证书
5. 系统获取真实证书哈希
6. 后端调用智能合约执行只读验证
7. 合约返回 IPFS 哈希、发布者地址和时间戳
8. 后端结合数据库业务信息生成验证结果
9. 前端展示证书哈希、IPFS 地址、机构信息和上链时间
10. 系统记录本次验证日志

该流程用于证明证书是否已经被可信登记到区块链中。

## 9. 证书文件真实性验证流程

系统支持直接上传文件进行真实性验证。

具体流程如下：

1. 用户进入文件验真页面
2. 上传待验证证书文件
3. 前端向后端发送文件
4. 后端对文件重新计算 SHA256 哈希
5. 系统用该哈希在数据库中查找已登记证书
6. 若数据库中不存在对应记录，则说明文件不可信或已被篡改
7. 若数据库中存在记录但链上不存在该哈希，则说明证书尚未上链
8. 若数据库和链上都存在一致记录，则判定文件真实
9. 前端显示文件哈希、链上哈希和判定结果
10. 系统写入验证日志

该流程会输出以下三类结果：

- 真实
- 未上链
- 被篡改

## 10. 跨机构授权验证流程

为了避免验证结果被无控制地明文暴露，系统设计了授权码机制。

### 10.1 授权码生成流程

1. 总管理员登录系统
2. 进入授权码管理页面
3. 设置授权码有效时长
4. 前端向后端发送生成请求
5. 后端校验当前用户是否为总管理员
6. 系统生成随机授权码
7. 旧授权码失效，仅保留最新授权码有效
8. 授权码哈希和过期时间写入数据库
9. 系统返回授权码明文供管理员分发

该流程的作用是为跨机构验证建立可控授权机制。

### 10.2 授权验证流程

1. 外部验证方执行证书验证或文件验真
2. 请求中可附带授权码
3. 后端校验授权码是否正确且在有效期内
4. 若授权成功，则返回明文验证结果
5. 若授权失败或未提供授权码，则返回加密后的结果
6. 前端展示验证状态
7. 系统记录日志

该流程保证系统在实现可验证性的同时，兼顾结果可见性的控制。

## 11. 验证日志记录流程

系统对重要验证行为进行日志留痕。

具体流程如下：

1. 用户完成链上验证或文件真实性验证
2. 前端将证书哈希和验证结果提交到日志接口
3. 后端将日志写入验证日志表
4. 系统保存验证结果和验证时间等信息
5. 后续可用于审计、统计和追踪分析

该流程为系统提供了基本的审计留痕能力。

## 12. 总结

本系统建立了一套较完整的电子证书可信管理流程。系统支持从用户注册登录、证书录入、草稿审批、正式发布、IPFS 存储和链上登记，到后续链上验证、文件验真、跨机构授权验证和日志审计的完整业务闭环。整体流程清晰，角色边界明确，能够较好体现区块链、IPFS 与传统 Web 系统融合应用的设计思路。
