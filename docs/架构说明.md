# 架构说明

## 1. 项目定位

本项目是一个面向电子证书场景的跨机构可信验证平台，采用“前端应用 + 后端服务 + 智能合约 + IPFS”的混合架构，实现用户注册登录、证书提交与发布、链上登记、文件真实性验证、跨机构授权验证和审计留痕。

从系统设计角度看，本项目并不是纯链上 DApp，而是以中心化业务后端为治理核心、以区块链承担可信索引与防篡改校验职责的业务系统。这种架构既有利于展示区块链技术的可信价值，又能够保证系统在权限控制、业务流转和数据管理方面具备较强的可实现性。

## 2. 当前整体架构

### 2.1 分层结构

系统当前可分为四个协作层：

1. **前端层**
   - 技术：React 18 + Vite 5 + ethers 6
   - 职责：页面交互、表单录入、OCR 调用、钱包连接、链上写入、结果展示

2. **后端层**
   - 技术：Node.js + Express 4 + MySQL
   - 职责：认证鉴权、证书业务处理、数据库读写、OCR 代理、IPFS 上传、链上只读验证、日志写入

3. **存储层**
   - MySQL：保存用户、证书、草稿、验证码、授权码、验证日志
   - IPFS：保存原始证书文件和 OCR 元数据

4. **链上层**
   - 技术：Solidity + Hardhat + ethers
   - 职责：保存证书哈希、IPFS 哈希、发布者地址、时间戳

### 2.2 协作关系

- 前端开发环境通过 Vite 代理把 `/api` 转发到 `http://localhost:3001`
- 前端调用后端接口处理注册、登录、列表查询、草稿审批、OCR、文件下载等中心化业务
- 前端在证书发布成功后，通过 MetaMask 调用合约完成 `storeCertificate`
- 后端通过 RPC 调用合约 `verifyCertificate` 执行链上只读验证
- 后端将原始证书文件与 OCR 元数据上传到 Pinata / IPFS
- MySQL 保存业务主数据，区块链只保存可信索引

## 3. 核心设计思路

### 3.1 链上只存索引

当前合约为每份证书保存五项数据：

- `certHash`
- `ipfsHash`
- `issuer`
- `timestamp`
- `revoked`

同时合约引入了基于 `owner` 和 `authorizedIssuers` 的访问控制机制，仅授权发行人可写入链上记录。

这样设计的主要依据如下：

- 降低链上存储成本
- 保持合约结构清晰，便于演示与维护
- 通过链上白名单控制写入权限，复杂业务流程仍留在后端
- 保留证书文件摘要的不可篡改能力
- 支持证书撤销标记，增强证书生命周期管理

### 3.2 前端负责上链

证书上链写入由浏览器侧通过 MetaMask 发起，而不是由后端代签。这样做的主要原因是：

- 用户显式确认交易
- 后端不保存任何用户私钥
- 更符合常见 Web3 交互方式
- 便于展示“业务发布”和“链上登记”两个阶段

### 3.3 保留中心化后端

当前后端承担的能力包括：

- 注册登录与 JWT 认证
- 邮箱验证码发送与校验
- 普通用户草稿提交
- 机构管理员草稿审批
- 证书查询、修改、删除
- OCR 调用与文本整合
- Pinata / IPFS 上传
- 链上只读验证
- 跨机构授权控制
- 验证日志留痕

这部分能力如果完全链上化，不仅成本较高，而且会显著增加实现复杂度与使用门槛。因此，本系统采用“链下业务治理 + 链上可信索引”的混合模式，以在工程可行性与可信性之间取得平衡。

### 3.4 架构层面的答辩要点

从毕业设计答辩角度，本架构可重点概括为以下几点：

- 采用分层设计，职责边界清晰
- 采用链上最小存储策略，控制区块链成本
- 采用链下业务治理策略，保证权限与流程可控
- 采用 IPFS 存储原始文件，降低链上压力
- 采用草稿审批机制，使系统更接近真实机构业务场景

## 5. 各模块职责

### 5.1 frontend

前端当前是一个单页应用，核心职责包括：

- 登录、注册、资料态管理
- 邮箱验证码发送
- 证书信息录入
- OCR 自动识别与表单回填
- 普通用户提交发布草稿
- 管理员直接发布证书
- 管理员审批草稿后继续上链
- 证书列表查询与管理
- 链上验证与文件真实性验证
- MetaMask 钱包连接与交易提交
- 总管理员生成跨机构授权码

前端已将部分功能拆分为独立组件（位于 `src/components/`），包括登录页面、侧边栏、草稿列表、授权码面板和管理员弹窗等；链上交互配置、钱包工具、OCR 解析和通用辅助函数已提取到 `src/utils/` 目录。

### 5.2 backend

后端是系统业务中枢，核心职责包括：

- 提供 `/api/users`、`/api/certificates`、`/api/verify-auth` 路由
- 解析并校验 JWT
- 注册、登录、获取当前用户资料
- 动态检查数据库表和字段
- 普通用户提交 `certificate_drafts`
- 管理员发布 `certificates`
- 上传原始证书文件到 IPFS
- 上传 OCR 元数据 JSON 到 IPFS
- 调用百度 OCR 识别图片文字
- 通过 RPC 校验证书链上记录
- 根据授权决定返回明文还是加密结果
- 记录验证日志

### 5.3 contracts

合约工程负责：

- 维护 `CertificateRegistry` 注册表合约
- 编译、测试与部署合约
- 输出 ABI 给前端或后端使用
- 保证同一 `certHash` 不能重复写入
- 提供单条和批量登记、撤销、扩展查询与存在性检测接口
- 通过 `owner` 与 `authorizedIssuers` 实现链上写入权限控制

## 7. 当前业务流程

### 7.1 注册与登录流程

1. 用户先调用发送邮箱验证码接口
2. 后端写入 `email_verify_codes` 并通过 SMTP 发送邮件
3. 用户提交用户名、密码、邮箱、验证码、机构和角色完成注册
4. 后端校验验证码并写入 `users`
5. 登录成功后后端签发 JWT
6. 前端后续请求通过 `Authorization: Bearer <token>` 携带登录态

### 7.2 证书发布流程

从业务组织方式看，系统将证书发布拆分为两种路径：

#### 普通用户流程

1. 普通用户填写证书信息并上传文件
2. 后端校验用户机构信息
3. 后端将文件和业务信息写入 `certificate_drafts`
4. 返回 `mode: draft`
5. 所属机构管理员在草稿箱中审批
6. 审批成功后由后端正式发布证书
7. 前端再发起链上写入并回写交易哈希

#### 管理员直接发布流程

1. 管理员上传文件并填写证书信息
2. 后端计算文件 SHA256 哈希
3. 后端检查 `certificates` 中是否已存在相同 `cert_hash`
4. 后端上传原始文件到 IPFS
5. 若存在 OCR 文本，后端同时上传 OCR 元数据 JSON 到 IPFS
6. 后端将证书记录写入 `certificates`
7. 前端连接 MetaMask 并调用合约 `storeCertificate`
8. 链上交易确认后调用接口保存 `tx_hash`

### 7.3 草稿审批流程

1. 机构管理员获取本机构 `pending` 草稿
2. 选择某条草稿执行审批
3. 后端先把草稿状态改为 `processing`
4. 后端复用正式发布逻辑生成证书记录
5. 后端回写草稿的 `approved_by`、`approved_at`、`published_cert_id`、`published_cert_hash`
6. 前端继续执行链上登记并保存交易哈希

### 7.4 证书链上验证流程

1. 输入值可以是数据库 `id`、`cert_id` 或真实 `cert_hash`
2. 后端优先在 `certificates` 中映射出真实 `cert_hash`
3. 后端通过 RPC 调用合约 `verifyCertificate`
4. 查询到链上记录后补充机构名称、钱包地址、认证状态
5. 若带有有效授权码则返回明文
6. 若无授权码则返回加密载荷

### 7.5 文件真实性验证流程

1. 后端对上传文件计算 SHA256
2. 优先用该哈希直接查链上
3. 若链上未命中，再遍历数据库中的 `cert_hash`
4. 同时兼容旧版“文件 + cert_id + cert_name + owner_name + issuer”组合哈希算法
5. 最终返回：
   - `真实`
   - `未上链`
   - `被篡改`

### 7.6 跨机构授权验证流程

1. 总管理员生成授权码
2. 验证请求通过请求头或请求体传入授权码
3. 后端先比对固定密钥，再校验数据库中的最新有效授权码
4. 授权通过则返回明文验证结果
5. 授权失败则返回 AES-GCM 加密结果

## 8. 数据存储分工

### 8.1 MySQL

当前代码显式依赖以下业务表：

- `users`
- `certificates`
- `certificate_drafts`
- `verify_logs`
- `verify_auth_codes`
- `email_verify_codes`

另有一个可选依赖表：

- `organizations`
  - 用于在链上验证结果中补充机构名称、钱包地址和认证状态
  - 如果不存在，不影响主流程

### 8.2 IPFS

IPFS 当前保存两类数据：

- 原始证书文件
- OCR 元数据 JSON

### 8.3 区块链

区块链当前保存：

- 证书哈希
- IPFS 哈希
- 发布者钱包地址
- 上链时间
- 撤销标记

## 9. 权限模型

系统当前使用三类角色：

- `user`
  - 可以注册、登录、提交证书草稿、查看个人可见数据

- `admin`
  - 可以查看本机构草稿
  - 可以审批本机构草稿
  - 可以管理本机构证书
  - 可以直接发布证书

- `super_admin`
  - 具备全局证书管理权限
  - 可以直接发布证书
  - 可以生成跨机构验证授权码

当前权限边界以应用层判断为主，数据库层没有建立强依赖外键。

## 10. 运行与部署关注点

### 10.1 前端

- 开发命令中内联了 `VITE_CERT_REGISTRY_ADDRESS`
- 通过 Vite 代理访问后端
- 证书上链依赖浏览器安装 MetaMask

### 10.2 后端

- 默认端口为 `3001`
- 启动时会自动创建部分表并补齐部分字段
- 依赖 SMTP、Pinata、百度 OCR、Sepolia RPC 等外部配置

### 10.3 合约

- 采用 Hardhat 编译部署
- 默认面向 Sepolia 网络
- 后端和前端都会读取合约地址配置

## 11. 当前实现特点

### 11.1 优点

- 技术链完整，覆盖前端、后端、数据库、区块链、IPFS、OCR
- 链上与链下职责划分清晰，设计思路明确
- 草稿审批机制使业务流程更接近真实机构场景
- 验证接口具备授权控制和加密返回能力
- 系统具备较强的展示性，适合用于毕业设计答辩说明

### 11.2 当前局限

- `App.jsx` 仍然较大，虽然已拆分出部分组件和工具模块，但主业务逻辑耦合度仍偏高
- `certificateController.js` 职责集中，后端分层不够细
- `users` 与 `certificates` 仍依赖手工建表或外部初始化
- 缺少前后端自动化测试和 migration 体系（合约已有单元测试覆盖）

## 12. 后续优化方向

- 继续拆分前端 `App.jsx` 中的业务逻辑，进一步抽取组件和 hooks
- 为后端引入 service / repository 分层
- 建立正式 migration 体系，而不是继续只靠运行时补字段
- 为前后端补充自动化测试
- 增加上传限制、限流、审计和异常告警能力
