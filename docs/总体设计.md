# 电子证书跨机构可信验证平台系统总体设计

## 1. 系统总体设计概述

系统总体设计是连接需求分析与系统实现的关键环节，其主要任务是根据前文提出的功能需求和非功能需求，确定系统的整体结构、模块划分、数据流转方式和关键技术实现思路，为后续数据库设计、智能合约设计和功能实现提供统一指导。

本系统围绕电子证书可信发布与跨机构验证场景展开，采用“前端应用 + 业务后端 + MySQL 数据库存储 + IPFS 文件存储 + 区块链可信索引”的混合架构。系统并非纯链上应用，而是在保留传统 Web 系统管理能力的基础上，引入区块链和分布式存储技术，用于增强电子证书关键索引数据的可信性和防篡改能力。

从设计目标上看，本系统不仅需要满足证书的发布与管理需求，还需要同时兼顾以下要求：

- 用户身份认证与权限控制
- 机构场景下的草稿审批机制
- 原始文件的分布式存储
- 关键索引的链上可信登记
- 按证书信息和文件两种方式进行真实性验证
- 跨机构验证中的结果访问控制
- 验证行为的审计留痕

因此，系统总体设计需要同时处理链下业务治理与链上可信验证的协同问题。

## 2. 系统设计目标

结合项目背景与需求分析，系统总体设计应达到以下目标：

### 2.1 功能完整性目标

系统应能够覆盖电子证书从生成、提交、审批、发布、上链、验证到审计的完整业务流程，形成闭环式管理能力。

### 2.2 可信性目标

系统应借助区块链不可篡改特性，对证书核心索引进行可信登记，并通过文件哈希比对和链上查询增强真实性验证能力。

### 2.3 可管理性目标

系统应保留中心化后端管理能力，实现用户认证、角色控制、草稿审批、授权控制和数据查询等业务治理功能。

### 2.4 可扩展性目标

系统应具备后续扩展空间，包括证书类别扩展、机构治理扩展、链上字段扩展和系统模块拆分优化等。

### 2.5 工程可实现性目标

系统总体设计应避免过度复杂化，在毕业设计场景下保证技术可落地、模块清晰、运行稳定、便于展示和答辩说明。

## 3. 系统总体架构设计

### 3.1 架构组成

本系统总体上划分为五个协作部分：

1. 前端表现层
2. 后端业务层
3. 数据库存储层
4. 分布式文件存储层
5. 区块链可信索引层

各层职责如下：

#### 3.1.1 前端表现层

前端基于 React 与 Vite 构建，主要负责：

- 登录注册界面展示
- 表单输入与数据交互
- OCR 识别结果展示和回填
- 草稿箱和证书列表页面展示
- 文件上传与验证结果展示
- MetaMask 钱包连接
- 链上交易发起与交互确认

前端是用户与系统交互的直接入口，承担页面交互和部分链上交互职责。

#### 3.1.2 后端业务层

后端基于 Node.js 与 Express 构建，主要负责：

- 用户注册登录和 JWT 鉴权
- 角色权限控制
- 证书发布与草稿审批业务
- 数据库读写
- OCR 服务调用
- IPFS 上传处理
- 链上只读验证
- 授权码校验
- 日志记录

后端是整个系统的业务中枢，负责将用户请求转化为具体的数据操作与业务处理流程。

#### 3.1.3 数据库存储层

系统采用 MySQL 保存链下业务数据，主要包括：

- 用户信息
- 正式证书信息
- 草稿证书信息
- 邮箱验证码
- 授权码
- 验证日志

数据库主要承担完整业务数据保存、状态流转记录和查询支撑任务。

#### 3.1.4 分布式文件存储层

原始证书文件和 OCR 元数据存储在 IPFS 中。后端通过 Pinata 网关将文件上传到 IPFS，并获取对应的 `ipfsHash`，供后续链上登记和前端访问使用。

#### 3.1.5 区块链可信索引层

系统使用 Solidity 编写智能合约，在区块链上保存电子证书的关键索引，包括：

- 证书哈希
- IPFS 哈希
- 发布者地址
- 上链时间

该层不承载复杂业务逻辑，只提供可信登记和查询能力。

### 3.2 分层协作关系

系统各层之间的协作关系如下：

1. 用户通过前端提交业务请求
2. 前端调用后端接口处理注册、登录、发布、审批、验证等业务
3. 后端将业务主数据写入 MySQL
4. 后端将原始文件和 OCR 元数据上传到 IPFS
5. 前端使用 MetaMask 调用智能合约，将证书哈希和 IPFS 哈希写入区块链
6. 后端在验证场景中通过 RPC 调用合约执行只读查询
7. 系统整合链上和链下数据后返回给前端展示

这种协作模式能够形成清晰的数据流和职责边界。

## 4. 系统功能模块设计

根据需求分析，系统主要划分为以下功能模块：

### 4.1 用户管理模块

该模块主要负责：

- 用户注册
- 邮箱验证码发送
- 用户登录
- 当前用户信息获取
- JWT 登录态维护

该模块为所有业务功能提供合法身份基础。

### 4.2 证书发布模块

该模块主要负责：

- 证书表单录入
- OCR 识别辅助录入
- 原始文件上传
- 普通用户草稿提交
- 管理员正式发布
- 交易哈希回写

该模块是系统最核心的业务入口。

### 4.3 草稿审批模块

该模块主要负责：

- 草稿列表查询
- 本机构草稿过滤
- 草稿状态控制
- 草稿审批后转为正式证书

该模块用于模拟真实机构内部的审核发布流程。

### 4.4 证书管理模块

该模块主要负责：

- 正式证书分页查询
- 条件检索
- 证书详情查看
- 证书信息修改
- 证书删除

该模块支撑证书全生命周期管理。

### 4.5 链上验证模块

该模块主要负责：

- 按 `id`、`cert_id`、`cert_hash` 等信息定位证书
- 调用智能合约进行链上只读验证
- 返回链上结果与链下补充信息

该模块体现系统的可信验证能力。

### 4.6 文件真实性验证模块

该模块主要负责：

- 上传待验证文件
- 重新计算文件 SHA256 哈希
- 比对数据库与链上记录
- 输出真实、未上链和被篡改结果

该模块增强了系统对电子证书文件本身的验证能力。

### 4.7 跨机构授权模块

该模块主要负责：

- 总管理员生成授权码
- 设置授权码有效期
- 验证请求中的授权码校验
- 控制验证结果明文或加密返回

该模块体现系统在可信验证基础上的受控共享设计。

### 4.8 日志审计模块

该模块主要负责：

- 记录链上验证日志
- 记录文件验真日志
- 保存验证结果和验证时间

该模块为系统后续追踪、统计和审计提供依据。

## 5. 核心业务流程设计

### 5.1 用户注册与登录流程

用户首先通过邮箱验证码完成注册，注册成功后使用用户名和密码进行登录。后端在登录成功后签发 JWT 令牌，前端在后续请求中携带该令牌进行认证。该流程实现了用户合法身份确认和登录态管理。

### 5.2 普通用户草稿发布流程

普通用户上传证书信息与原始文件后，系统不会直接生成正式证书，而是将相关数据写入草稿表，等待所属机构管理员审批。这一流程体现了机构场景下的业务审核机制。

### 5.3 管理员直接发布流程

管理员上传证书信息和文件后，后端先计算文件哈希，再检查是否存在重复证书。若校验通过，则将原始文件上传至 IPFS，将证书业务信息写入数据库，随后由前端调用 MetaMask 完成链上登记。

### 5.4 草稿审批发布流程

管理员在草稿箱中查看待审批数据，对符合要求的草稿执行确认发布。系统将草稿状态改为处理中后，复用正式发布逻辑生成正式证书，并最终更新草稿状态为已审批。

### 5.5 链上验证流程

用户输入证书编号、数据库 id 或证书哈希后，后端将其解析为真实证书哈希，然后调用智能合约执行链上验证，读取证书对应的 IPFS 哈希、发布者地址和上链时间，并将结果返回给前端展示。

### 5.6 文件真实性验证流程

用户上传待验证的证书文件后，后端重新计算文件哈希，先查数据库，再查链上记录，最终输出真实、未上链或被篡改等结果。该流程为电子证书文件提供了独立于编号查询的验证方式。

### 5.7 授权控制流程

在跨机构验证场景中，总管理员可生成具有有效期的授权码。验证请求若携带有效授权码，则返回明文结果；否则仅返回加密后的数据，从而实现结果访问范围控制。

## 6. 数据设计思路

### 6.1 链下数据设计思路

系统链下数据主要保存在 MySQL 中，原因如下：

- 业务数据结构化程度高
- 需要支持条件查询和分页管理
- 需要支持事务与状态流转
- 需要保存完整业务信息和审计数据

链下主要数据包括用户、正式证书、证书草稿、验证码、授权码和验证日志。

### 6.2 链上数据设计思路

系统链上仅保存关键索引数据，而不保存完整业务数据。这样设计的主要原因如下：

- 降低链上存储成本
- 简化合约结构
- 提高开发和验证效率
- 保留证书关键信息的防篡改能力

### 6.3 文件存储设计思路

原始证书文件和 OCR 元数据不适合直接上链，因此采用 IPFS 存储。IPFS 哈希既可作为文件定位标识，也可与链上索引建立关联，形成完整的数据协同路径。

## 7. 权限控制设计

本系统采用角色分层权限控制机制，共分为三类角色：

### 7.1 普通用户

主要权限包括：

- 注册登录
- 上传证书图片和文件
- 提交证书草稿
- 查询和验证证书

普通用户不具备正式发布和审批权限。

### 7.2 机构管理员

主要权限包括：

- 直接发布证书
- 查看本机构草稿
- 审批本机构草稿
- 查询、修改和删除本机构证书

### 7.3 总管理员

主要权限包括：

- 具备管理员全部能力
- 生成跨机构验证授权码
- 具备更大范围的数据管理能力

这种权限设计既满足基本业务需求，又使角色边界清晰，便于后续扩展。

## 8. 系统安全设计

为了提高系统在原型阶段的安全性与规范性，总体设计中考虑了以下安全措施：

- 用户密码采用 bcrypt 加密存储
- 使用 JWT 控制登录态和接口访问权限
- 普通用户、管理员和总管理员权限分离
- 后端不托管任何钱包私钥
- 验证结果在未授权时以加密形式返回
- 使用文件 SHA256 哈希作为证书内容唯一可信摘要
- 重复证书发布和重复链上写入均进行拦截

这些设计虽然并非生产级全量安全方案，但已经能够满足毕业设计原型系统的安全要求。

## 9. 系统设计特点分析

本系统总体设计具有以下特点：

### 9.1 链上链下职责分离明确

系统将复杂业务治理留在链下，将可信索引登记留在链上，形成了较清晰的职责边界。

### 9.2 业务流程贴近真实机构场景

通过普通用户草稿提交和管理员审批发布机制，系统较好模拟了机构内部审核流程。

### 9.3 可信索引与文件存储协同

系统采用“文件上 IPFS、索引上链”的设计方式，在可信性和成本之间取得平衡。

### 9.4 验证方式多样化

既支持根据证书编号或证书哈希进行链上验证，也支持直接上传文件进行真实性验证，增强了系统实用性。

### 9.5 具备受控共享思路

通过授权码机制，系统能够根据权限控制验证结果的明文和加密返回方式，体现了跨机构数据共享中的安全考虑。

## 10. 总体设计局限性与后续优化方向

虽然本系统总体设计已经能够支撑毕业设计原型实现，但仍存在一些局限性：

- 前端已拆分出部分组件和工具模块，但 `App.jsx` 主业务逻辑仍偏集中
- 后端控制器承担职责较多，可继续按服务层拆分
- 当前权限模型仍以应用层判断为主，数据库层约束较弱
- 智能合约已支持撤销和访问控制，但尚未支持版本管理和更细粒度的链上机构治理
- 前后端缺少自动化测试（合约已有单元测试覆盖）

后续优化方向包括：

- 继续拆分前端业务逻辑，后端引入 service 分层
- 增加更细粒度的组织与权限模型
- 扩展智能合约版本管理和链上治理能力
- 为前后端补充自动化测试和部署规范
- 优化安全机制和异常治理能力

## 11. 本章小结

本章从系统目标、总体架构、功能模块、业务流程、数据设计、权限控制和安全设计等方面对电子证书跨机构可信验证平台进行了总体设计说明。整体设计采用混合架构模式，将链下业务治理、IPFS 文件存储和区块链可信索引有机结合，既满足了电子证书管理与验证的业务需求，也体现了区块链技术在可信存证场景中的应用价值。

上述总体设计为后续数据库设计、智能合约设计和系统功能实现提供了结构性依据，也为论文后续章节展开奠定了基础。
