# 电子证书跨机构可信验证平台核心功能实现说明

## 1. 文档概述

本文档用于说明电子证书跨机构可信验证平台中各项核心功能的具体实现方式，重点从系统实现角度分析用户认证、证书发布、草稿审批、链上登记、真实性验证、授权控制和日志留痕等关键模块。本文档适合作为毕业论文中“系统功能实现”或“核心模块实现”章节的基础内容。

## 2. 系统实现思路

本系统采用前后端分离与链上链下协同的实现方式：

- 前端负责页面交互、表单录入、OCR 识别结果展示、MetaMask 钱包连接和验证结果展示
- 后端负责认证鉴权、业务逻辑处理、数据库读写、IPFS 上传、OCR 调用、链上只读验证和日志记录
- MySQL 负责保存链下主业务数据
- IPFS 负责保存原始证书文件和 OCR 元数据
- 区块链智能合约负责保存证书可信索引

系统通过“链下治理 + 链上存证”的方式完成电子证书可信管理，既保留了传统业务系统的权限控制能力，又引入了区块链不可篡改的可信特性。

## 3. 用户注册与登录功能实现

### 3.1 邮箱验证码发送实现

用户注册前，系统要求先完成邮箱验证码校验。后端在收到邮箱验证码发送请求后，首先校验邮箱格式是否合法，并检查短时间内是否重复发送。若满足条件，则生成验证码，进行哈希处理后写入 `email_verify_codes` 表，并通过 SMTP 服务发送邮件。

该实现方式具有以下特点：

- 防止无效邮箱注册
- 控制验证码发送频率
- 验证码仅保存哈希值，不直接保存明文
- 验证码具备时效限制，降低被重复利用风险

### 3.2 用户注册实现

用户注册时需要提交用户名、密码、邮箱、邮箱验证码、机构和角色等信息。后端首先检查用户名与邮箱是否已存在，然后校验验证码是否有效且未过期。校验通过后，系统使用 bcrypt 对密码进行加密，并将用户信息写入 `users` 表。

这一实现方式保证了：

- 账号唯一性
- 密码安全存储
- 注册行为建立在邮箱可验证基础上
- 后续可基于角色和机构实现权限控制

### 3.3 用户登录实现

用户登录时输入用户名和密码，后端查询数据库中的用户记录，并使用 bcrypt 比较密码哈希值。校验成功后，系统生成 JWT 令牌，并将用户基本信息与令牌一并返回前端。前端保存令牌，并在后续需要认证的接口请求中通过请求头携带该令牌。

该功能实现后，系统可以完成：

- 用户身份确认
- 登录态管理
- 基于 JWT 的接口访问控制

## 4. 证书发布功能实现

证书发布功能是本系统最核心的业务模块，按照角色分为普通用户草稿提交流程和管理员直接发布流程。

### 4.1 OCR 辅助识别实现

为了减少用户在填写证书信息时的手工输入工作量，系统提供 OCR 辅助录入能力。用户上传证书图片后，前端将图片提交给后端 OCR 接口，后端调用百度 OCR 服务完成文本识别，并将识别结果返回前端。前端进一步对识别文本进行字段提取，回填到证书表单中。

OCR 功能的实现特点如下：

- 支持从证书图片中提取常见字段
- 识别结果以辅助方式参与录入，不强制覆盖用户输入
- 允许用户对识别结果进行手工修正
- 提高证书录入效率，降低人工填写错误率

### 4.2 普通用户草稿提交实现

当登录用户角色为普通用户时，系统不允许其直接发布正式证书，而是将其提交内容写入草稿表。用户在前端填写证书信息并上传原始文件后，后端识别其角色为 `user`，随后检查其所属机构是否存在。若机构信息完整，则将证书信息和文件二进制数据写入 `certificate_drafts` 表，并将状态设置为 `pending`。

该实现方式具有以下优点：

- 体现真实业务中的审批机制
- 防止普通用户绕过审核直接发布
- 保留完整草稿信息，便于后续审批和转正
- 为管理员集中处理草稿提供数据基础

### 4.3 管理员直接发布实现

当登录用户角色为管理员或总管理员时，系统允许直接发布正式证书。其主要实现步骤如下：

1. 前端提交证书表单及原始文件
2. 后端读取文件内容并计算 SHA256 哈希
3. 检查数据库中是否已存在相同证书哈希
4. 若不存在重复，则上传原始文件到 IPFS
5. 若存在 OCR 文本，则同时上传 OCR 元数据 JSON 到 IPFS
6. 将证书业务信息写入 `certificates` 表
7. 返回证书哈希、IPFS 哈希和证书标识信息给前端

这里使用文件哈希作为证书内容唯一摘要，其核心价值在于：

- 保证同一证书内容可以被唯一标识
- 为后续链上登记和文件真实性验证提供统一依据
- 防止重复内容被重复发布

### 4.4 交易哈希回写实现

管理员直接发布成功后，前端需要继续调用 MetaMask 发起区块链交易。在用户钱包确认交易后，前端会获得链上交易哈希，然后再通过后端接口将交易哈希写回数据库中的 `tx_hash` 字段。

这样实现的意义在于：

- 将链下业务记录与链上交易建立关联
- 便于后续查看某条证书的链上登记状态
- 为审计和追踪提供完整证据链

## 5. 草稿审批功能实现

### 5.1 草稿列表查询实现

管理员进入草稿箱后，前端会请求后端获取待审批草稿数据。后端首先验证当前用户是否为机构管理员，然后根据管理员所属机构过滤 `certificate_drafts` 表中的数据，仅返回该机构且状态为 `pending` 的草稿列表。

这一实现方式保证了：

- 草稿审批范围按机构隔离
- 管理员只能处理本机构数据
- 草稿流程符合组织管理逻辑

### 5.2 草稿审批发布实现

管理员点击“确认发布”后，后端并不是直接简单修改状态，而是先将草稿状态从 `pending` 改为 `processing`，防止多个管理员重复处理同一草稿。随后系统读取草稿中的证书信息和原始文件，复用正式发布逻辑执行以下操作：

- 计算文件哈希
- 检查重复证书
- 上传 IPFS
- 写入正式证书表

发布成功后，再将草稿状态更新为 `approved`，并回写正式证书 id 和证书哈希。

该实现方式具有以下特点：

- 支持审批过程中的并发保护
- 避免同一草稿被重复发布
- 复用正式发布逻辑，减少重复代码
- 保证草稿与正式证书之间的映射关系可追踪

## 6. 证书查询与管理功能实现

系统为登录用户提供证书列表查询与管理功能。前端支持按证书编号、持证人、证书类别和证书哈希等条件进行搜索。后端在接收到查询请求后，会根据当前登录用户的角色和所属机构决定数据可见范围。

具体实现策略如下：

- 总管理员具备全局范围管理权限
- 普通管理员默认仅查看本机构相关证书
- 查询结果支持分页返回
- 管理员可进一步执行修改和删除操作

该模块实现后，系统能够完成正式证书的全生命周期管理，而不仅仅局限于证书发布。

## 7. 链上登记功能实现

### 7.1 前端钱包交互实现

证书发布完成后，前端通过 ethers 连接浏览器中的 MetaMask 钱包，并调用合约中的 `storeCertificate` 方法，将证书哈希和 IPFS 哈希写入区块链。用户需要在钱包中手动签名并确认交易。

该设计的核心思想是：

- 由用户自己控制钱包私钥
- 后端不托管任何私钥信息
- 链上写入行为具有用户确认过程
- 更符合区块链应用的交互习惯

### 7.2 链上登记与链下数据协同实现

链上只负责记录最关键的可信索引，而完整业务数据仍保存在数据库中。也就是说：

- 原始文件保存在 IPFS
- 业务详细信息保存在 MySQL
- 证书哈希和 IPFS 哈希上链保存

这一实现策略形成了“链上索引 + 链下业务 + 分布式文件”的协同结构，使系统在可信性、成本和可维护性之间取得平衡。

## 8. 链上验证功能实现

证书链上验证功能支持用户通过数据库主键、业务证书编号或证书哈希进行核验。前端提交验证请求后，后端首先解析输入值，并将其转换为真实证书哈希。随后，后端通过以太坊 RPC 调用智能合约的 `verifyCertificate` 方法，读取链上记录。

后端在获得链上返回的 IPFS 哈希、发布者地址和时间戳后，还会进一步补充链下信息，例如：

- 证书哈希
- 机构名称
- 钱包地址
- 认证状态
- 上链时间

最终再返回给前端展示。

该功能实现的意义在于：

- 验证证书是否确实被登记到区块链
- 验证链上索引是否与链下业务记录一致
- 为外部验证方提供可信核验依据

## 9. 文件真实性验证功能实现

文件真实性验证功能是本系统区别于普通证书查询系统的重要能力之一。其实现逻辑如下：

1. 用户上传待验证的证书文件
2. 后端重新计算该文件的 SHA256 哈希
3. 使用该哈希在数据库中查找匹配证书
4. 若数据库无匹配记录，则认为文件不可信或已被篡改
5. 若数据库中存在匹配记录，但链上找不到对应哈希，则判定为未上链
6. 若数据库中存在匹配记录且链上也能验证通过，则判定文件真实

系统最终会返回以下三种结果之一：

- 真实
- 未上链
- 被篡改

这一模块的价值在于：

- 用户无需先知道证书编号即可验证文件
- 可直接通过文件本身判断真实性
- 有助于发现内容被修改后的伪造文件

## 10. 跨机构授权验证功能实现

### 10.1 授权码生成实现

为了控制跨机构验证结果的可见范围，总管理员可以在前端授权码管理页面设置授权码有效期，并向后端发起生成请求。后端校验当前用户身份为 `super_admin` 后，生成随机授权码，将其哈希值与有效期写入 `verify_auth_codes` 表，同时使历史有效授权码失效，仅保留最新一个有效授权码。

这种实现机制具有以下优点：

- 避免明文授权码直接持久化存储
- 限制授权码生效窗口
- 保持授权策略简单清晰
- 适合毕业设计中的原型验证场景

### 10.2 明文与加密结果分流实现

在链上验证和文件真实性验证接口中，后端会优先检查请求中是否携带有效授权码：

- 若授权码有效，则直接返回明文验证结果
- 若授权码无效或未提供，则返回加密后的数据结构

系统采用该策略的主要目的在于：

- 支持跨机构受控共享
- 避免未经授权方直接看到完整验证信息
- 在保证可验证性的同时增强隐私控制能力

## 11. 验证日志留痕功能实现

系统在完成链上验证或文件真实性验证后，会由前端向日志接口发送证书哈希和验证结果，后端将其写入 `verify_logs` 表，并记录验证来源和验证时间。该功能可用于：

- 后续验证行为审计
- 异常访问追踪
- 验证次数统计
- 系统运行过程留痕

日志模块虽然不直接参与证书可信判断，但对系统管理和论文展示都具有重要意义。

## 12. 核心实现特点总结

综合来看，本系统核心功能实现具有以下特点：

- 采用 JWT 完成登录鉴权，保证接口访问安全
- 采用邮箱验证码机制增强注册可信性
- 采用普通用户草稿、管理员审批的发布模式，贴近真实机构业务
- 采用文件 SHA256 哈希作为证书可信标识，统一发布和验证逻辑
- 采用 IPFS 存储原始文件，降低链上存储压力
- 采用 MetaMask 钱包完成上链确认，避免后端托管私钥
- 采用链上验证和文件验证双模式，提升实用性
- 采用授权码控制跨机构验证结果可见范围
- 采用日志机制提升系统可审计性
