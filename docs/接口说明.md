# 接口说明

## 1. 接口总览

本接口文档用于说明系统后端对外提供的主要业务能力。后端统一以 `/api` 作为前缀，当前接口分为五类：

- 用户接口：注册、登录、资料读取、邮箱验证码
- 证书接口：发布、草稿、列表、验证、OCR、下载、管理
- 授权接口：生成跨机构验证授权码
- 日志接口：写入验证日志
- 健康检查接口：服务与数据库状态检查

从毕业设计答辩角度，这些接口共同支撑了“用户认证—证书流转—链上登记—受控验证—审计留痕”的完整业务闭环。

## 2. 通用约定

### 2.1 基础地址

- 开发时前端通过 Vite 代理把 `/api` 转发到 `http://localhost:3001`
- 后端默认监听端口为 `3001`

### 2.2 认证方式

需要登录的接口通过请求头携带 JWT：

```http
Authorization: Bearer <token>
```

### 2.3 验证授权传递方式

链上验证和文件验证支持通过以下方式传入跨机构授权码：

- 请求头 `x-verify-authorization`
- 请求头 `x-cross-verify-auth`
- 请求头 `authorization_verify`
- 请求体 `verify_authorization`
- 请求体 `authorization`

### 2.4 通用返回结构

接口整体采用如下风格：

```json
{
  "ok": true,
  "message": "操作成功",
  "data": {},
  "pagination": {}
}
```

常见字段说明：

- `ok`：请求是否成功
- `message`：结果说明
- `data`：业务数据
- `pagination`：分页信息
- `mode`：发布接口返回的业务流转模式
- `encrypted`：验证结果是否为加密返回

### 2.5 文件上传

以下接口需要使用 `multipart/form-data`：

- `POST /api/certificates/publish`
- `POST /api/certificates/verify-file`
- `POST /api/certificates/ocr`

## 3. 用户接口

### 3.1 发送邮箱验证码

- 方法：`POST`
- 路径：`/api/users/send-email-code`
- 认证：否

请求体：

```json
{
  "email": "name@example.com"
}
```

说明：

- 邮箱格式不正确时返回 `400`
- 发送过于频繁时返回 `429`
- 邮箱已注册时返回 `409`
- 成功后验证码有效期为 5 分钟，发送冷却 60 秒

成功响应示例：

```json
{
  "ok": true,
  "message": "验证码已发送，请查收邮箱",
  "data": {
    "email": "name@example.com",
    "ttlMinutes": 5,
    "cooldownSeconds": 60
  }
}
```

### 3.2 用户注册

- 方法：`POST`
- 路径：`/api/users/register`
- 认证：否

请求体：

```json
{
  "username": "test",
  "password": "123456",
  "email": "name@example.com",
  "emailCode": "123456",
  "organization": "某机构",
  "role": "user"
}
```

说明：

- `username`、`password`、`email`、`emailCode` 为必填
- `role` 不传时默认为 `user`
- 后端会先检查用户名和邮箱是否已存在
- 注册成功后会删除该邮箱对应的验证码记录

成功响应示例：

```json
{
  "ok": true,
  "message": "注册成功",
  "data": {
    "id": 1,
    "username": "test",
    "role": "user",
    "email": "name@example.com",
    "organization": "某机构"
  }
}
```

### 3.3 用户登录

- 方法：`POST`
- 路径：`/api/users/login`
- 认证：否

请求体：

```json
{
  "username": "test",
  "password": "123456"
}
```

成功响应示例：

```json
{
  "ok": true,
  "message": "登录成功",
  "data": {
    "token": "jwt-token",
    "user": {
      "id": 1,
      "username": "test",
      "role": "user",
      "email": "name@example.com",
      "organization": "某机构"
    }
  }
}
```

### 3.4 获取当前用户资料

- 方法：`GET`
- 路径：`/api/users/profile`
- 认证：是

成功响应示例：

```json
{
  "ok": true,
  "data": {
    "id": 1,
    "username": "test",
    "role": "user",

  }

```

## 4. 证书接口

### 4.1 发布证书

- 方法：`POST`
- 路径：`/api/certificates/publish`
- 认证：是
- 内容类型：`multipart/form-data`

表单字段：

- `cert_name`：证书名称，必填
- `owner_name`：持证人，必填
- `issuer`：发证机构，必填
- `cert_no`：证书编号，可选
- `issue_date`：发证日期，可选
- `cert_category` 或 `certCategory`：证书类别，可选
- `ocr_text` 或 `ocrText`：OCR 文本，可选
- `file`：证书文件，必填

说明：

- 普通用户调用时不会直接发布，而是进入草稿流程
- 管理员和总管理员调用时会直接发布
- 后端会计算文件 SHA256，并检查是否已存在重复 `cert_hash`
- 若配置了 OCR 文本，会额外上传 OCR 元数据到 IPFS

普通用户响应示例：

```json
{
  "ok": true,
  "mode": "draft",
  "message": "草稿已提交，待所属机构管理员确认后发布",
  "data": {
    "id": 12,
    "cert_name": "学位证书",
    "owner_name": "张三",
    "issuer": "某大学",
    "cert_no": "1001",
    "cert_category": "学历",
    "issue_date": "2026-04-01",
    "status": "pending",
    "submitted_organization": "某大学"
  }
}
```

管理员直接发布响应示例：

```json
{
  "ok": true,
  "mode": "published",
  "message": "证书发布成功",
  "data": {
    "id": 1,
    "cert_no": "1001",
    "cert_id": "uuid",
    "cert_name": "学位证书",
    "owner_name": "张三",
    "issuer": "某大学",
    "issue_date": "2026-04-01",
    "cert_category": "学历",
    "ocr_text": "识别文本",
    "ocr_ipfs_hash": "QmXXX",
    "ocr_chain_key": "ocr:sha256hash",
    "cert_hash": "sha256hash",
    "ipfs_hash": "QmYYY"
  }
}
```

### 4.2 获取草稿列表

- 方法：`GET`
- 路径：`/api/certificates/drafts`
- 认证：是
- 权限：仅 `admin`

查询参数：

- `page`
- `pageSize`
- `owner_name` 或 `ownerName`
- `cert_no` 或 `certNo`
- `status`

说明：

- 默认只查询 `pending` 草稿
- 仅返回当前管理员所属机构的草稿
- 管理员没有机构信息时会返回 `400`

成功响应示例：

```json
{
  "ok": true,
  "data": [
    {
      "id": 12,
      "draft_id": "uuid",
      "cert_name": "学位证书",
      "owner_name": "张三",
      "issuer": "某大学",
      "cert_no": "1001",
      "issue_date": "2026-04-01",
      "cert_category": "学历",
      "status": "pending",
      "submitted_by": 8,
      "submitted_organization": "某大学",
      "approved_by": null,
      "approved_at": null,
      "published_cert_id": null,
      "created_at": "2026-04-01T00:00:00.000Z",
      "updated_at": "2026-04-01T00:00:00.000Z"
    }
  ],
  "pagination": {
    "page": 1,
    "pageSize": 10,
    "total": 1
  }
}
```

### 4.3 审批草稿并发布

- 方法：`POST`
- 路径：`/api/certificates/drafts/:id/approve`
- 认证：是
- 权限：仅 `admin`

说明：

- 仅能审批本机构 `pending` 草稿
- 后端审批时会先把草稿锁定为 `processing`
- 审批成功后会生成正式证书记录，并回写 `published_cert_id` 和 `published_cert_hash`
- 审批完成后仍需要前端继续发起链上登记并回写交易哈希

成功响应示例：

```json
{
  "ok": true,
  "message": "草稿确认发布成功",
  "data": {
    "id": 21,
    "cert_no": "1001",
    "cert_id": "uuid",
    "cert_name": "学位证书",
    "owner_name": "张三",
    "issuer": "某大学",
    "issue_date": "2026-04-01",
    "cert_category": "学历",
    "cert_hash": "sha256hash",
    "ipfs_hash": "QmYYY"
  }
}
```

### 4.4 保存交易哈希

- 方法：`PATCH`
- 路径：`/api/certificates/:id/tx`
- 认证：是

请求体：

```json
{
  "tx_hash": "0x123",
  "cert_id": "uuid"
}
```

说明：

- 证书成功上链后调用
- 优先按路径参数 `id` 更新
- 若按 `id` 未命中且传入了 `cert_id`，后端会再按 `cert_id` 更新
- 当前 `certificates` 表缺少 `tx_hash` 时会返回 `400`

### 4.5 查询证书列表

- 方法：`GET`
- 路径：`/api/certificates`
- 认证：是

查询参数：

- `page`
- `pageSize`

- `cert_no` 或 `certNo`
- `owner_name` 或 `ownerName`
- `cert_category` 或 `certCategory`
- `cert_hash` 或 `certHash`

说明：

- 支持分页和多条件筛选
- `super_admin` 拥有全局范围
- 非全局角色默认按机构范围过滤
- 当表缺少某些扩展字段时，后端会按现有字段兼容返回

成功响应示例：

```json
  "data": [
```

### 4.6 管理员修改证书

- 方法：`PATCH`
- 路径：`/api/certificates/:id`
- 认证：是
- 权限：`admin` 或 `super_admin`

请求体示例：

```json
{
  "cert_name": "更新后的证书名称",
  "owner_name": "张三",
  "issuer": "某大学",
  "cert_no": "1001",
```

### 4.7 管理员删除证书

- 方法：`DELETE`
- 路径：`/api/certificates/:id`
- 认证：是
- 权限：`admin` 或 `super_admin`

说明：

- `super_admin` 可全局删除
- 普通管理员仅能删除本机构证书

### 4.8 链上验证证书

- 方法：`GET`
- 路径：`/api/certificates/verify/:certHash`
- 认证：否
- 授权：可选

输入说明：

- 路径参数虽然命名为 `certHash`
- 实际支持：
  - 数据库 `id`
  - `cert_id`
  - 真实 `cert_hash`

后端处理逻辑：

- 先在数据库中查出真实 `cert_hash`
- 再通过 RPC 调用合约只读验证
- 若存在 `organizations` 表且匹配钱包地址，则补充机构信息
- 授权成功返回明文
- 未授权返回加密载荷

明文返回示例：

```json
{
  "ok": true,
  "encrypted": false,
  "data": {
    "cert_hash": "sha256hash",
    "ipfs_hash": "QmYYY",
    "issuer": "0xabc",
    "timestamp": 1710000000,
    "organizationName": "某大学",
    "walletAddress": "0xabc",
    "status": "已认证"
  }
}
```

加密返回示例：

```json
{
  "ok": true,
  "encrypted": true,
  "message": "未获得跨机构验证授权，已返回加密内容",
  "data": {
    "nonce": "base64",
    "iv": "base64",
    "tag": "base64",
    "ciphertext": "base64"
  }
}
```

### 4.9 文件真实性验证

- 方法：`POST`
- 路径：`/api/certificates/verify-file`
- 认证：否
- 授权：可选
- 内容类型：`multipart/form-data`

表单字段：

- `file`

结果说明：

- `真实`：文件哈希对应链上已登记证书
- `未上链`：数据库中存在匹配记录，但链上没有对应记录
- `被篡改`：链上和数据库均无法匹配

明文返回示例：

```json
{
  "ok": true,
  "encrypted": false,
  "data": {
    "fileHash": "sha256hash",
    "chainHash": "sha256hash",
    "result": "真实"
  }
}
```

### 4.10 OCR 识别

- 方法：`POST`
- 路径：`/api/certificates/ocr`
- 认证：否
- 内容类型：`multipart/form-data`

表单字段：

- `file`

说明：

- 当前后端调用百度 OCR 通用文字识别
- 若未识别到有效文字，会返回 `422`
- 若 OCR 服务异常，通常返回 `502`

成功响应示例：

```json
{
  "ok": true,
  "data": {
    "text": "证书名称\n张三\n某大学",
    "words": ["证书名称", "张三", "某大学"]
  }
}
```

### 4.11 下载证书文件

- 方法：`GET`
- 路径：`/api/certificates/download/:ipfsHash`
- 认证：否

说明：

- 后端会依次尝试多个 IPFS 网关
- 成功时返回附件流
- 文件名格式为 `certificate-<ipfsHash>.<ext>`

## 5. 授权接口

### 5.1 生成跨机构验证授权码

- 方法：`POST`
- 路径：`/api/verify-auth/codes`
- 认证：是
- 权限：仅 `super_admin`

请求体：

```json
{
  "ttlMinutes": 60
}
```

说明：

- `ttlMinutes` 不传时默认 `60`
- 最小值为 `1` 分钟，最大值为 `7` 天
- 新授权码生成后，旧的有效授权码会立即失效
- 数据库存储的是哈希，不存储明文授权码

成功响应示例：

```json
{
  "ok": true,
  "message": "授权码生成成功",
  "data": {
    "code": "AUTH-XXXXXXXX",
    "ttlMinutes": 60,
    "expiresAt": "2026-04-01T00:00:00.000Z",
    "policy": "latest_only"
  }
}
```

## 6. 日志接口

### 6.1 写入验证日志

- 方法：`POST`
- 路径：`/api/verify/log`
- 认证：否

请求体：

```json
{
  "cert_hash": "sha256hash",

}
```

说明：

- `cert_hash` 为必填
- `result` 不传时默认写入 `成功`
- 后端会自动记录 `x-forwarded-for` 或当前请求 IP

成功响应示例：

```json
{
  "ok": true,
  "message": "验证日志记录成功",
  "data": {
    "id": 1,
    "cert_hash": "sha256hash",
    "verifier_ip": "::1",
    "verify_time": "2026-04-01T00:00:00.000Z",
    "result": "成功"
  }
}
```

## 9. 答辩说明与注意事项

- 验证接口在未授权时会返回加密结构而不是明文，这体现了系统对敏感信息披露范围的控制
- `drafts` 相关接口仅机构管理员可用，不开放给普通用户和总管理员，这体现了多角色分层权限设计
- `publish` 接口会根据当前角色自动进入草稿模式或直接发布模式，这体现了业务流程与角色权限的联动关系
- 当前文档基于现有代码整理，部分字段会因数据库兼容逻辑按实际表结构降级返回
