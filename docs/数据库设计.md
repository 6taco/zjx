# 数据库设计

## 1. 设计目标

本项目数据库用于承载电子证书平台的链下业务数据、权限控制信息与审计数据，是系统实现业务治理、状态留痕与权限约束的核心基础设施。数据库设计目标包括：

- 支撑用户注册、登录和角色权限判断
- 支撑普通用户提交流程与机构管理员审批流程
- 支撑证书发布、检索、修改、删除与链上映射
- 支撑邮箱验证码和跨机构授权码机制
- 支撑验证日志留痕
- 与区块链、IPFS 共同形成“链上索引 + 链下业务”的混合架构

从毕业设计答辩角度看，该数据库设计体现了“链下保存完整业务数据、链上保存关键索引数据”的总体思路，有利于说明系统如何在可信性、可管理性和实现成本之间取得平衡。

## 2. 数据库与连接配置

后端当前采用 MySQL 连接池访问数据库，代码中的默认配置为：

- `DB_HOST` 默认 `localhost`
- `DB_PORT` 默认 `3306`
- `DB_USER` 默认 `root`
- `DB_PASSWORD` 默认 `123456`
- `DB_NAME` 默认 `bishe`

连接池策略：

- `waitForConnections: true`
- `connectionLimit: 10`
- `queueLimit: 0`

说明：

- 上述默认值来自当前代码兜底，仅适合本地开发演示
- 生产环境应通过环境变量覆盖，不应继续使用默认口令

## 3. 当前总体数据模型

### 3.1 当前代码依赖表

当前代码显式依赖 6 张业务表：

- `users`
- `certificates`
- `certificate_drafts`
- `verify_logs`
- `verify_auth_codes`
- `email_verify_codes`

### 3.2 可选依赖表

当前还有一张可选依赖表：

- `organizations`
  - 仅用于在链上验证结果中补充机构名称、钱包地址和认证状态
  - 缺失时不会影响发布、验证、登录等主流程

### 3.3 数据职责划分

- `users`：用户账号、角色、邮箱、机构
- `certificate_drafts`：普通用户待审批证书草稿
- `certificates`：正式发布证书
- `verify_logs`：验证行为日志
- `verify_auth_codes`：跨机构授权码
- `email_verify_codes`：邮箱验证码

## 4. 启动期 Schema 保障机制

后端启动时会自动执行以下表结构保障逻辑：

- 创建 `verify_logs`
- 创建 `verify_auth_codes`
- 创建 `email_verify_codes`
- 创建 `certificate_drafts`
- 检查 `certificates` 是否存在
- 若 `certificates` 已存在，则补齐以下字段：
  - `cert_no`
  - `issue_date`
  - `cert_category`
  - `ocr_text`
  - `ocr_ipfs_hash`

说明：

- `users` 不会自动创建
- `certificates` 主表不会自动创建，只会在表已存在时补字段
- 这意味着部署前仍需手工初始化 `users` 和 `certificates`

这一设计在答辩中可以表述为：系统采用“关键辅助表自动保障、核心主表人工确认”的初始化策略，以兼顾演示便利性与主数据可控性。

## 5. 表结构设计

### 5.1 users

用途：保存账号、登录凭证、角色与机构信息。

代码显式依赖字段：

- `id`
- `username`
- `password`
- `role`
- `email`
- `organization`
- `created_at`
- `updated_at`

主要读写场景：

- 注册时按 `username` 和 `email` 查重
- 注册成功后插入用户记录
- 登录时按 `username` 查询
- 获取当前用户资料时按 `id` 查询
- 生成授权码或发布证书时按 `id` 查询角色和机构

设计建议：

- `username` 建唯一索引
- `email` 建唯一索引
- `role` 建议使用受控枚举：`user`、`admin`、`super_admin`
- `password` 必须保存 bcrypt 哈希

### 5.2 certificates

用途：保存正式发布后的证书业务主数据。

代码显式依赖字段：

- 主键与标识：
  - `id`
  - `cert_id`
  - `cert_no`
- 业务字段：
  - `cert_name`
  - `owner_name`
  - `issuer`
  - `issue_date`
  - `cert_category`
- 可信字段：
  - `cert_hash`
  - `ipfs_hash`
  - `tx_hash`
- OCR 字段：
  - `ocr_text`
  - `ocr_ipfs_hash`
- 发证关系字段：
  - `issuer_user_id`
  - `issuer_id`
- 审计字段：
  - `created_at`
  - `updated_at`

主要读写场景：

- 管理员直接发布证书
- 草稿审批通过后写入正式证书
- 按 `id`、`cert_id`、`cert_no`、`owner_name`、`cert_category`、`cert_hash` 分页查询
- 按 `id` / `cert_id` / `cert_hash` 映射链上验证输入
- 更新 `tx_hash`
- 管理员修改和删除
- 文件真实性验证时匹配 `cert_hash`

兼容机制：

- 代码会在运行时探测列是否存在
- 查询语句根据现有列动态拼接
- 因此旧表结构在一定范围内仍可运行，但能力会退化

索引建议：

- 唯一索引：`cert_id`
- 唯一索引：`cert_hash`
- 普通索引：`cert_no`
- 普通索引：`owner_name`
- 普通索引：`cert_category`
- 普通索引：`created_at`
- 组合索引：`issuer_user_id, created_at`

### 5.3 certificate_drafts

用途：保存普通用户提交、等待机构管理员审批的证书草稿。

当前表由后端启动时自动创建，字段包括：

- `id` BIGINT AUTO_INCREMENT
- `draft_id` VARCHAR(64) NOT NULL
- `cert_name` VARCHAR(255) NOT NULL
- `owner_name` VARCHAR(255) NOT NULL
- `issuer` VARCHAR(255) NOT NULL
- `cert_no` VARCHAR(128) NULL
- `issue_date` DATE NULL
- `cert_category` VARCHAR(64) NULL
- `ocr_text` LONGTEXT NULL
- `file_name` VARCHAR(255) NULL
- `file_mime` VARCHAR(128) NULL
- `file_data` LONGBLOB NOT NULL
- `submitted_by` BIGINT NOT NULL
- `submitted_organization` VARCHAR(255) NULL
- `status` VARCHAR(32) NOT NULL DEFAULT 'pending'
- `approved_by` BIGINT NULL
- `approved_at` DATETIME NULL
- `published_cert_id` BIGINT NULL
- `published_cert_hash` VARCHAR(128) NULL
- `created_at`
- `updated_at`

主要状态：

- `pending`
- `processing`
- `approved`

主要读写场景：

- 普通用户发布时写入草稿
- 管理员按机构查看草稿列表
- 管理员审批时先锁定草稿，再发布正式证书
- 审批完成后回写审批人与正式证书信息

已建索引：

- 唯一键：`uniq_certificate_drafts_draft_id (draft_id)`
- 索引：`idx_certificate_drafts_status_created (status, created_at)`
- 索引：`idx_certificate_drafts_submitted_by (submitted_by)`
- 索引：`idx_certificate_drafts_submitted_org (submitted_organization)`

### 5.4 verify_logs

用途：记录验证行为，支持审计与统计。

字段：

- `id` BIGINT AUTO_INCREMENT
- `cert_hash` VARCHAR(128) NOT NULL
- `verifier_ip` VARCHAR(128) NOT NULL
- `verify_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
- `result` VARCHAR(32) NOT NULL

主要读写场景：

- 前端在验证后调用日志接口写入
- 后续可用于查看验证频率、验证来源和结果统计

已建索引：

- `idx_verify_logs_cert_hash (cert_hash)`
- `idx_verify_logs_verify_time (verify_time)`

### 5.5 verify_auth_codes

用途：保存跨机构验证授权码哈希和有效期。

字段：

- `id` BIGINT AUTO_INCREMENT
- `code_hash` VARCHAR(128) NOT NULL
- `expires_at` DATETIME NOT NULL
- `created_by` BIGINT NULL
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

主要读写场景：

- 总管理员生成授权码时插入新记录
- 新码生成时把旧有效记录批量置为失效
- 验证时只读取最新且未过期的一条记录进行比对

已建索引与约束：

- 唯一键：`uniq_verify_auth_code_hash (code_hash)`
- 索引：`idx_verify_auth_expires_at (expires_at)`
- 索引：`idx_verify_auth_created_by (created_by)`

安全特性：

- 只保存授权码 SHA256 哈希
- 不保存明文授权码

### 5.6 email_verify_codes

用途：保存注册邮箱验证码哈希和过期时间。

字段：

- `id` BIGINT AUTO_INCREMENT
- `email` VARCHAR(255) NOT NULL
- `code_hash` VARCHAR(128) NOT NULL
- `expires_at` DATETIME NOT NULL
- `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP

主要读写场景：

- 发送验证码时插入新记录
- 注册时读取最新未过期验证码
- 注册成功后删除该邮箱全部验证码记录

已建索引：

- `idx_email_verify_email_created (email, created_at)`
- `idx_email_verify_expires_at (expires_at)`

安全特性：

- 只保存验证码 SHA256 哈希
- 不保存明文验证码

### 5.7 organizations

用途：可选机构信息表，仅用于增强链上验证回显。

当前代码会探测该表是否存在，并优先读取以下字段：

- `wallet_address`
- `org_name`
- `status`

说明：

- 若表不存在，验证接口仍会正常工作
- 若存在但缺少 `wallet_address`，当前增强查询也会自动退化

## 6. 逻辑关系与权限关系

### 6.1 逻辑关系

- `users (1) -> (N) certificate_drafts`
  - 通过 `submitted_by`

- `users (1) -> (N) certificates`
  - 通过 `issuer_user_id` 或兼容字段表达

- `users (1) -> (N) verify_auth_codes`
  - 通过 `created_by`

- `certificates (1) -> (N) verify_logs`
  - 通过 `cert_hash`

### 6.2 当前约束特点

- 关系主要由应用层保证，不依赖强外键
- 权限也主要由代码层按角色和机构判断
- 当前库结构强调兼容性，弱化了数据库层的严格约束

## 7. 查询模式与性能关注

### 7.1 高频查询模式

- 用户登录按 `username` 精确查询
- 用户注册按 `username` 和 `email` 查重
- 草稿列表按 `status + submitted_organization` 查询
- 证书列表按机构范围 + 条件分页查询
- 链上验证前按 `id`、`cert_id`、`cert_hash` 映射证书
- 日志按 `cert_hash` 与 `verify_time` 检索

### 7.2 当前性能风险点

- 文件真实性验证在链上未命中后，会全量读取 `certificates` 再遍历比对
- `certificate_drafts.file_data` 直接保存二进制文件，草稿表体积会快速增长
- 草稿审批和正式发布存在双存储路径，需关注数据库容量

### 7.3 优化建议

- 为 `cert_hash`、`cert_id`、`cert_no` 保持稳定索引
- 后续将文件真实性验证改为更多精确匹配路径，减少全表遍历
- 为草稿文件考虑对象存储或临时文件存储方案，而不是长期放在 LONGBLOB
- 对日志、验证码、过期授权码建立定时清理机制

## 8. 安全设计

### 8.1 数据安全

- 用户密码使用 bcrypt 哈希存储
- 邮箱验证码和授权码使用 SHA256 哈希存储
- 无授权的验证请求只返回加密结果，不直接返回明文

### 8.2 配置安全

- 数据库、JWT、OCR、Pinata、RPC 等配置应通过环境变量提供
- 当前代码包含默认弱配置兜底，仅适合开发环境

### 8.3 审计能力

- `verify_logs` 提供最基础的验证留痕
- 当前尚未覆盖管理员修改、删除、审批等操作日志

## 9. 答辩要点与后续演进建议

### 9.1 可用于答辩说明的设计要点

- 数据库负责保存完整业务状态，区块链只负责保存关键索引，职责分工明确
- 草稿表与正式证书表分离，能够体现审批前后业务状态的差异
- 授权码与邮箱验证码均采用哈希存储，体现了基础安全设计意识
- 通过运行时探测与补字段机制兼容历史表结构，体现了系统演进适配能力

### 9.2 后续演进建议

### 9.2.1 结构演进

- 引入正式 migration 工具统一管理表结构版本
- 明确 `users`、`certificates`、`certificate_drafts` 的外键关系
- 为正式证书引入软删除字段

### 9.2.2 存储演进

- 将草稿文件从 `LONGBLOB` 迁移到对象存储或 IPFS 临时区
- 为大文本 OCR 内容和日志表制定归档策略

### 9.2.3 安全演进

- 使用最小权限数据库账号
- 清理默认弱口令和默认密钥
- 为高频验证接口增加限流与风控

## 10. 建议 DDL 草案

以下 DDL 用于表达当前代码期望的数据结构，可作为初始化或迁移参考。

说明：

- `users` 和 `certificates` 为建议初始化主表
- `verify_logs`、`verify_auth_codes`、`email_verify_codes`、`certificate_drafts` 虽然代码可自动创建，但统一用 DDL 初始化会更清晰
- `organizations` 为可选增强表

```sql
CREATE TABLE IF NOT EXISTS users (
  id BIGINT NOT NULL AUTO_INCREMENT,
  username VARCHAR(128) NOT NULL,
  password VARCHAR(255) NOT NULL,
  role VARCHAR(64) NOT NULL DEFAULT 'user',
  email VARCHAR(255) NOT NULL,
  organization VARCHAR(255) NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uniq_users_username (username),
  UNIQUE KEY uniq_users_email (email),
  INDEX idx_users_role_org (role, organization)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE IF NOT EXISTS certificates (
  id BIGINT NOT NULL AUTO_INCREMENT,
  cert_id VARCHAR(128) NOT NULL,
  cert_no VARCHAR(128) NULL,
  cert_name VARCHAR(255) NOT NULL,
  owner_name VARCHAR(255) NOT NULL,
  issuer VARCHAR(255) NOT NULL,
  cert_hash VARCHAR(128) NOT NULL,
  ipfs_hash VARCHAR(255) NOT NULL,
  tx_hash VARCHAR(128) NULL,
  issue_date DATE NULL,
  cert_category VARCHAR(64) NULL,
  ocr_text LONGTEXT NULL,
  ocr_ipfs_hash VARCHAR(255) NULL,
  issuer_user_id BIGINT NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uniq_certificates_cert_id (cert_id),
  UNIQUE KEY uniq_certificates_cert_hash (cert_hash),
  INDEX idx_certificates_cert_no (cert_no),
  INDEX idx_certificates_owner_name (owner_name),
  INDEX idx_certificates_cert_category (cert_category),
  INDEX idx_certificates_issuer_user_id_created_at (issuer_user_id, created_at),
  INDEX idx_certificates_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE IF NOT EXISTS certificate_drafts (
  id BIGINT NOT NULL AUTO_INCREMENT,
  draft_id VARCHAR(64) NOT NULL,
  cert_name VARCHAR(255) NOT NULL,
  owner_name VARCHAR(255) NOT NULL,
  issuer VARCHAR(255) NOT NULL,
  cert_no VARCHAR(128) NULL,
  issue_date DATE NULL,
  cert_category VARCHAR(64) NULL,
  ocr_text LONGTEXT NULL,
  file_name VARCHAR(255) NULL,
  file_mime VARCHAR(128) NULL,
  file_data LONGBLOB NOT NULL,
  submitted_by BIGINT NOT NULL,
  submitted_organization VARCHAR(255) NULL,
  status VARCHAR(32) NOT NULL DEFAULT 'pending',
  approved_by BIGINT NULL,
  approved_at DATETIME NULL,
  published_cert_id BIGINT NULL,
  published_cert_hash VARCHAR(128) NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uniq_certificate_drafts_draft_id (draft_id),
  INDEX idx_certificate_drafts_status_created (status, created_at),
  INDEX idx_certificate_drafts_submitted_by (submitted_by),
  INDEX idx_certificate_drafts_submitted_org (submitted_organization)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE IF NOT EXISTS verify_logs (
  id BIGINT NOT NULL AUTO_INCREMENT,
  cert_hash VARCHAR(128) NOT NULL,
  verifier_ip VARCHAR(128) NOT NULL,
  verify_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  result VARCHAR(32) NOT NULL,
  PRIMARY KEY (id),
  INDEX idx_verify_logs_cert_hash (cert_hash),
  INDEX idx_verify_logs_verify_time (verify_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE IF NOT EXISTS verify_auth_codes (
  id BIGINT NOT NULL AUTO_INCREMENT,
  code_hash VARCHAR(128) NOT NULL,
  expires_at DATETIME NOT NULL,
  created_by BIGINT NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uniq_verify_auth_code_hash (code_hash),
  INDEX idx_verify_auth_expires_at (expires_at),
  INDEX idx_verify_auth_created_by (created_by)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE IF NOT EXISTS email_verify_codes (
  id BIGINT NOT NULL AUTO_INCREMENT,
  email VARCHAR(255) NOT NULL,
  code_hash VARCHAR(128) NOT NULL,
  expires_at DATETIME NOT NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  INDEX idx_email_verify_email_created (email, created_at),
  INDEX idx_email_verify_expires_at (expires_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

CREATE TABLE IF NOT EXISTS organizations (
  id BIGINT NOT NULL AUTO_INCREMENT,
  org_name VARCHAR(255) NULL,
  wallet_address VARCHAR(255) NOT NULL,
  status VARCHAR(64) NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  UNIQUE KEY uniq_organizations_wallet_address (wallet_address)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```
