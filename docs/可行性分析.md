# 电子证书跨机构可信验证平台可行性分析

## 1. 可行性分析概述

可行性分析是系统开发前的重要论证环节，其主要作用在于判断项目是否具备实施条件，以及是否能够在现有技术、成本和应用场景下达到预期目标。对于电子证书跨机构可信验证平台而言，可行性分析不仅需要关注常规 Web 系统开发条件，还需要综合考虑区块链、IPFS、OCR、数据库和前后端协同实现的可操作性。

本项目拟解决电子证书在发布、验证和跨机构共享过程中的真实性难保障、验证效率低以及数据可信性不足等问题。系统采用“前端应用 + 后端服务 + MySQL 数据库存储 + IPFS 文件存储 + 区块链可信索引”的混合技术架构，通过链下业务治理与链上可信登记相结合的方式实现业务目标。

从项目实现情况看，系统已完成用户注册登录、邮箱验证码校验、证书草稿提交流程、管理员审批发布、证书文件上传 IPFS、智能合约上链登记、链上验证、文件真实性验证、跨机构授权验证和验证日志记录等核心功能，说明该项目在整体方案上具备较强的实现基础。因此，有必要从技术、经济、操作和应用等角度对本系统进行系统性可行性分析。

## 2. 技术可行性分析

### 2.1 整体技术路线可行性

本项目采用前后端分离与链上链下协同的实现模式，整体技术路线清晰，所选技术栈成熟，适合毕业设计和原型系统开发。

系统当前技术组成如下：

- 前端：React + Vite + ethers
- 后端：Node.js + Express
- 数据库：MySQL
- 智能合约：Solidity + Hardhat
- 去中心化存储：IPFS（通过 Pinata 接入）
- 钱包交互：MetaMask
- OCR 服务：百度 OCR

上述技术均具有较成熟的开发生态和丰富的资料支持，能够有效降低开发门槛和调试难度。其中，React 和 Node.js 适合快速构建交互式 Web 系统；MySQL 适合管理结构化业务数据；Solidity 与 Hardhat 适合实现智能合约原型；IPFS 能够承担原始文件存储任务；MetaMask 能够完成浏览器侧链上交易签名；OCR 服务能够为证书信息提取提供辅助支持。

因此，从技术栈选择角度看，本项目具备明确且可落地的实现路径。

### 2.2 前端实现可行性

系统前端采用 React 和 Vite 构建，具有组件化开发、响应式交互和开发效率高等特点，适合本项目多功能页面的实现需求。当前前端已集成以下主要功能：

- 注册与登录界面
- 邮箱验证码发送
- 证书发布表单
- OCR 图片识别入口
- 草稿箱管理
- 证书列表查询与管理
- 链上验证页面
- 文件真实性验证页面
- 授权码管理页面

前端还通过 ethers 与 MetaMask 完成交互，使用户能够在浏览器中直接完成区块链交易确认，而不需要后端托管私钥。这种方式既符合常见 Web3 应用交互模式，也增强了私钥管理的安全性。

从项目实现情况看，前端模块已经能够较完整地支撑业务流程，因此前端实现方案具有较强可行性。

### 2.3 后端实现可行性

系统后端采用 Node.js 与 Express 搭建，具有开发效率高、接口组织灵活、与前端 JavaScript 技术栈统一等优点，适合本项目的快速构建需求。当前后端已实现以下关键能力：

- 用户注册、登录和资料获取
- JWT 认证与权限校验
- 证书发布与草稿提交
- 草稿审批发布
- 证书列表查询、修改和删除
- OCR 调用代理
- IPFS 文件上传
- 链上只读验证
- 跨机构授权校验
- 验证日志写入

后端通过控制器和路由划分功能模块，逻辑组织较清晰，能够满足本项目功能实现需求。同时，后端通过数据库连接池管理 MySQL 访问，并在启动时对部分辅助表执行自动保障，降低了运行环境初始化难度。

从代码实现和工程结构来看，后端已具备稳定支撑主要业务流程的能力，因此后端技术方案可行。

### 2.4 数据库存储可行性

本项目使用 MySQL 存储链下业务数据。电子证书平台涉及用户、证书、草稿、验证码、授权码和验证日志等结构化数据，这些数据天然适合关系型数据库管理。MySQL 在以下方面满足系统需求：

- 支持结构化表设计，便于组织用户和证书数据
- 支持索引机制，便于根据证书编号、证书哈希等字段快速查询
- 支持事务，适合注册、草稿审批和授权码生成等需要一致性的场景
- 成熟稳定，部署和维护成本较低

当前系统中已设计并使用 `users`、`certificates`、`certificate_drafts`、`verify_logs`、`verify_auth_codes` 和 `email_verify_codes` 等数据表，已经能够覆盖主要业务需求。因此，从数据管理角度看，MySQL 作为链下业务数据存储方案是可行的。

### 2.5 区块链方案可行性

本项目采用 Solidity 编写智能合约，并通过 Hardhat 进行编译与部署，前端通过 MetaMask 发起链上交易，后端通过 RPC 完成只读链上验证。当前智能合约仅保存以下四类核心字段：

- 证书哈希
- IPFS 哈希
- 发布者地址
- 时间戳

这种设计遵循“链上只存关键索引”的原则，具有以下优点：

- 避免将大量原始文件直接写入链上，从而降低存储成本
- 保留证书关键摘要的不可篡改能力
- 简化合约结构，降低开发和调试难度
- 便于与链下数据库和 IPFS 形成分工协作

同时，合约中通过 `certHash` 唯一性约束避免同一证书重复登记，能够满足毕业设计原型系统对可信登记的基本要求。因此，从链上方案设计和实现复杂度角度看，该区块链方案是可行的。

### 2.6 IPFS 文件存储可行性

电子证书通常包含原始图片或文件，不适合直接存储到区块链中。IPFS 作为去中心化文件存储技术，适合承担原始证书文件和 OCR 元数据的存储任务。其可行性主要体现在以下方面：

- 能够通过内容寻址机制唯一标识文件
- 适合保存证书原始文件，减轻链上压力
- 可与区块链中的哈希索引形成关联
- 通过 Pinata 等网关服务可降低直接使用 IPFS 的复杂度

当前系统已实现将原始证书文件上传至 IPFS，并将返回的 IPFS 哈希与证书哈希一同用于后续展示和验证。因此，IPFS 在本系统中的应用具有现实可行性。

### 2.7 OCR 集成可行性

本项目引入 OCR 功能，主要目的是减少用户在录入证书信息时的手工输入成本。后端通过百度 OCR 接口识别证书图片中的文字，再由前端对识别结果进行提取和表单回填。从技术角度看，这种方式具有以下优点：

- OCR 服务接口成熟，可直接集成
- 可提取证书名称、持证人、编号、日期等常见字段
- 能作为辅助录入手段，而非唯一数据来源
- 即使识别结果不完全准确，用户仍可手动修正，降低误差风险

因此，OCR 功能以“辅助录入”而非“完全自动化录入”的方式接入系统，是具备可行性的。

### 2.8 技术风险与可控性分析

尽管本项目总体技术路线可行，但仍存在一定技术风险，主要包括：

- OCR 识别准确率受图片质量影响较大
- IPFS 上传依赖外部服务可用性
- 区块链交易存在确认延迟
- 测试网络稳定性不如生产环境
- 前后端和链上链下联动流程相对复杂

不过，这些风险整体可控，原因在于：

- OCR 仅作为辅助功能，不影响主业务闭环
- IPFS 与区块链异常均有明确错误提示
- 链上只写入少量关键字段，降低交互复杂度
- 采用测试网络进行开发验证，适合毕业设计原型场景
- 核心业务数据仍保存在后端数据库中，保证系统治理能力

因此，从技术风险可控性看，本项目具备实施可行性。

## 3. 经济可行性分析

### 3.1 开发成本分析

本项目主要用于毕业设计研究与原型实现，开发环境以个人计算机、本地数据库、测试网络和开源技术栈为基础，不需要大规模硬件采购和商业软件授权。主要成本构成包括：

- 个人开发设备
- 本地运行环境配置
- 测试网络使用
- IPFS 网关服务接入
- OCR 接口调用成本
- 域名和服务器成本（若后续部署到公网）

其中，React、Node.js、Express、MySQL、Hardhat 和 Solidity 等核心技术均为开源技术，无需支付额外授权费用；区块链开发可以基于测试网络完成；论文阶段通常也不要求投入正式生产级资源。因此，项目整体开发成本较低。

### 3.2 运行成本分析

系统在毕业设计阶段的运行成本主要集中在以下方面：

- 本地或轻量云主机运行前后端服务
- MySQL 数据库存储开销
- IPFS 文件存储服务开销
- OCR 接口按调用量产生的少量费用
- 测试网络下的链上交互成本较低

由于系统采用“文件上 IPFS、索引上链”的方式，避免了将大量原始数据写入区块链，从而显著降低了区块链存储成本。对毕业设计原型系统而言，该成本是可以接受的。

### 3.3 维护成本分析

系统维护主要包括：

- 后端接口维护
- 前端页面优化
- 数据库备份与管理
- 智能合约升级与重新部署
- OCR 与 IPFS 配置调整

由于系统采用常见的前后端分离结构，代码模块边界相对明确，后期维护难度总体适中。对于毕业设计而言，系统维护成本处于可控范围内。

### 3.4 经济可行性结论

本项目采用的技术栈以开源和测试环境为主，硬件和软件投入较小，运行成本和维护成本均处于较低水平。项目不依赖高成本商业化基础设施，适合作为毕业设计项目实施。因此，从经济角度看，本系统具备良好的可行性。

## 4. 操作可行性分析

### 4.1 用户操作可行性

系统面向的使用对象包括普通用户、机构管理员、总管理员和外部验证方。整体交互流程围绕注册、登录、发布、审批、验证等常见 Web 系统操作展开，用户学习成本较低。主要体现在：

- 登录注册流程符合一般网站使用习惯
- 发布证书通过表单填写方式完成，易于理解
- OCR 功能降低了录入工作量
- 草稿箱和证书列表界面便于管理员进行集中处理
- 验证模块支持按编号、哈希或文件验证，操作路径清晰

即使用户不具备深度区块链技术背景，也只需要在发布阶段通过 MetaMask 完成确认操作，因此整体可操作性较强。

### 4.2 管理流程可行性

系统在操作流程上符合现实机构管理逻辑。普通用户只能提交草稿，机构管理员负责审批发布，总管理员负责跨机构授权控制，体现了角色分层和责任划分。这种业务流程设计更接近实际管理场景，因此在组织使用上具有较强合理性。

### 4.3 部署与运行可行性

系统前后端运行环境要求较常规，部署流程相对清晰：

- 前端可通过 Vite 启动
- 后端可通过 Node.js 运行
- 数据库使用 MySQL
- 合约通过 Hardhat 部署
- 浏览器安装 MetaMask 即可完成钱包交互

在开发、演示和答辩环境中，该部署方式足够简单，适合毕业设计项目展示和运行验证。

### 4.4 操作可行性结论

从用户使用、管理员操作以及部署运行角度看，本系统流程清晰、上手门槛较低、运行条件明确，具备较好的操作可行性。
