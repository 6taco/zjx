# 电子证书跨机构可信验证平台项目文档

## 1. 项目简介

本项目是一个面向电子证书场景的跨机构可信验证平台，采用 **Web 前端 + 业务后端 + 智能合约 + IPFS** 的组合架构，实现用户认证、证书草稿提交、管理员审批发布、链上登记、链上核验、文件真伪校验与跨机构授权验证等核心能力。

从毕业设计答辩的角度看，本系统重点解决的是“电子证书如何实现可信发布、便捷验证与跨机构受控共享”这一问题。系统既保留了传统业务系统在权限控制和数据管理方面的优势，又利用区块链的不可篡改特性提升了关键索引数据的可信度。

项目目标包括：

- 为证书发布机构提供可管理、可追踪的电子证书发布能力
- 为证书持有者或验证方提供可验证、可追溯的证书真伪核验能力
- 通过链上登记提高关键索引数据的防篡改能力
- 通过授权码机制支持跨机构验证时的受控信息披露
- 通过 OCR 识别减少证书录入成本

## 2. 系统核心能力

### 2.1 用户与认证能力

- 用户注册、登录
- 邮箱验证码注册
- 当前用户资料读取
- 基于 JWT 的身份认证

### 2.2 证书业务能力

- 普通用户提交证书草稿
- 机构管理员查看并审批本机构草稿
- 管理员和总管理员直接发布证书
- 自动计算证书文件 SHA256 哈希
- 上传证书文件到 IPFS
- 写入 MySQL 保存业务信息
- 调用 MetaMask 将证书哈希与 IPFS 地址登记到智能合约
- 保存链上交易哈希

### 2.3 验证能力

- 按数据库 `id`、`cert_id`、`cert_hash` 进行链上验证
- 上传文件进行真伪校验
- 输出 `真实 / 未上链 / 被篡改`
- 记录验证日志

### 2.4 管理能力

- 管理员查看证书列表
- 管理员修改证书信息
- 管理员删除证书
- 总管理员生成跨机构验证授权码

### 2.5 智能辅助能力

- OCR 识别证书图片中的文字
- 自动提取证书名称、持证人、发证机构、证书编号、日期等字段
- 将 OCR 文本作为补充元数据存储到 IPFS

## 3. 总体架构设计

### 3.1 架构分层

系统当前可分为四层：

1. **前端层**
   - 基于 React + Vite
   - 负责用户交互、表单处理、草稿审批入口、验证页、MetaMask 钱包交互

2. **后端层**
   - 基于 Node.js + Express + MySQL
   - 负责用户认证、证书业务处理、数据库访问、Pinata/IPFS 上传、OCR 调用、验证日志记录、跨机构授权校验

3. **存储层**
   - MySQL 保存用户、证书、草稿、验证码、授权码、验证日志
   - IPFS 保存原始证书文件和 OCR 元数据

4. **链上层**
   - 基于 Solidity + Hardhat
   - 负责证书哈希与 IPFS 哈希的不可篡改登记与查询

### 3.2 关键设计思路

- 前端负责上链写入，链上交易由用户在浏览器中通过 MetaMask 签名确认
- 后端负责业务中枢能力，统一处理用户、证书、OCR、IPFS、权限与日志
- 合约保持最小化，只保存 `certHash`、`ipfsHash`、`issuer`、`timestamp`
- 证书发布支持“普通用户草稿模式”和“管理员直接发布模式”
- 跨机构验证在未授权时只返回加密内容，以降低敏感信息直接暴露风险
- 整体架构体现“链上存证、链下治理”的设计思路

## 8. 核心业务流程设计

### 8.1 用户与权限流程

1. 用户注册时填写用户名、密码、邮箱、机构、角色
2. 前端先请求发送邮箱验证码
3. 后端校验邮箱格式、发送频率和邮箱是否已注册
4. 注册时后端校验验证码并加密密码
5. 登录成功后后端签发 JWT
6. 前端后续请求携带 `Authorization: Bearer <token>`

角色分层：

- `user`：提交草稿、查看个人可见数据
- `admin`：查看并审批本机构草稿、直接发布证书、管理本机构证书
- `super_admin`：全局管理证书并生成跨机构授权码

### 8.2 证书发布流程

为适配真实业务场景，本系统将证书发布区分为两条路径：

#### 普通用户草稿流程

1. 用户上传文件和表单数据到 `/api/certificates/publish`
2. 后端识别当前角色为 `user`
3. 后端把证书信息和二进制文件写入 `certificate_drafts`
4. 返回 `mode: draft`
5. 机构管理员查看草稿并执行审批
6. 审批成功后后端生成正式证书记录
7. 前端继续发起 MetaMask 上链并回写 `tx_hash`

#### 管理员直接发布流程

1. 管理员上传文件和表单数据到 `/api/certificates/publish`
2. 后端计算 SHA256
3. 后端检查是否存在重复 `cert_hash`
4. 后端上传原始文件到 Pinata / IPFS
5. 若存在 OCR 文本，则把 OCR 元数据 JSON 上传到 IPFS
6. 后端把证书信息写入 `certificates`
7. 前端调用合约 `storeCertificate(cert_hash, ipfs_hash)`
8. 等待交易确认后调用接口保存 `tx_hash`

### 8.3 草稿审批流程

1. 机构管理员调用 `/api/certificates/drafts`
2. 后端只返回当前机构的 `pending` 草稿
3. 管理员调用 `/api/certificates/drafts/:id/approve`
4. 后端先把草稿状态置为 `processing`
5. 后端复用正式发布逻辑生成证书
6. 后端回写 `approved_by`、`approved_at`、`published_cert_id`、`published_cert_hash`

### 8.4 证书列表与管理流程

功能包括：

- 分页查询
- 按 `id`、`cert_id`、`cert_no`、持证人、类别、Hash 检索
- 下载证书文件
- 管理员修改证书信息
- 管理员删除证书

权限控制上，后端会根据当前用户所属机构限制数据范围，`super_admin` 拥有更大的查看与管理范围。这种设计有助于体现系统在多机构场景下的权限隔离能力。

### 8.5 链上验证流程

1. 用户输入数据库 `id`、`cert_id` 或 `cert_hash`
2. 后端先在数据库中映射出真实 `cert_hash`
3. 后端通过 RPC 调用合约 `verifyCertificate`
4. 若存在 `organizations` 表，则补充机构名称、钱包地址和认证状态
5. 若请求中提供有效授权码，则返回明文
6. 若未授权，则返回加密结果

### 8.6 文件真实性验证流程

1. 后端计算文件哈希
2. 先检查该哈希是否直接存在于链上
3. 若不存在，则读取数据库中的 `certificates`
4. 同时兼容旧版哈希计算路径
5. 输出结果：
   - `真实`
   - `未上链`
   - `被篡改`

### 8.7 OCR 流程

1. 用户上传证书图片
2. 后端调用百度 OCR
3. 返回识别出的文本与分词结果
4. 前端进一步解析文本并尝试抽取关键字段
5. 自动填充发布表单

### 8.8 跨机构授权验证流程

1. 总管理员生成时效性授权码
2. 验证请求通过请求头或请求体传入授权码
3. 后端优先校验固定授权密钥，再校验数据库中的最新有效授权码
4. 验证通过则返回明文数据
5. 验证失败则返回 AES-GCM 加密结果

## 9. 数据存储设计

### 9.1 MySQL

当前代码依赖以下业务表：

- `users`
- `certificates`
- `certificate_drafts`
- `verify_logs`
- `verify_auth_codes`
- `email_verify_codes`

其中：

- `users` 和 `certificates` 为主表，但不会被完全自动创建
- `certificate_drafts` 用于承载普通用户待审批数据
- `verify_logs` 用于记录验证行为
- `verify_auth_codes` 和 `email_verify_codes` 都只存哈希，不存明文

可选增强表：

- `organizations`
  - 用于链上验证结果补充机构名称、钱包地址和状态

### 9.2 IPFS

IPFS 中主要保存：

- 原始证书文件
- OCR 元数据 JSON

### 9.3 区块链

区块链中只保存最关键、最需要防篡改的数据：

- 证书哈希
- IPFS 哈希
- 发布者地址
- 上链时间

## 14. 当前项目的可改进点

### 14.1 前端方面

- `App.jsx` 体量较大，建议进一步拆分为多个页面组件、业务 hooks、工具函数

### 14.3 安全方面

- 当前代码包含开发期兜底密钥与地址，正式环境应全部替换为安全配置

### 14.4 工程化方面

- 当前缺少自动化测试
- 当前缺少更细粒度的前后端模块拆分
- 可以加入 ESLint、Prettier、单元测试与接口测试

## 15. 适合答辩展示的亮点总结

1. **区块链与传统 Web 架构结合**
   - 不是单纯 DApp，而是将链上可信与中心化业务处理结合

2. **跨机构可信验证机制**
   - 通过授权码与加密返回机制控制验证结果披露范围

本项目本质上是一个以“电子证书可信发布与验证”为核心场景的全栈系统。它并非单纯的链上应用，而是将前端交互能力、后端业务治理能力、MySQL 数据管理能力、IPFS 文件存储能力与区块链防篡改能力有机结合，形成了一套较完整的电子证书可信管理方案。
